1. Why Object-Oriented Programming?

Structure grows more important as a system becomes larger.

Without structure

One giant function takes orders, cooks food, serves customers, receives payment, and cleans the restaurant.

  • Too many responsibilities
  • Hard to test
  • Small changes create unexpected bugs
  • Logic becomes tightly coupled

With OOP

Separate objects represent a waiter, chef, cleaner, order, and payment.

  • Each object owns a clear responsibility
  • Data and related behavior stay together
  • Components can be extended and tested separately
  • Real-world entities are easier to model
Memory line: OOP organizes a system as collaborating objects, where each object owns data and behavior.
Interview perspective: Do not say OOP is always better. Small scripts and simple data transformations may be clearer with functions. OOP is most useful when the domain has identifiable entities, changing state, reusable behavior, and many interacting components.

2. Class and Object

A class is a blueprint; an object is a concrete instance.

Class

Defines what data and behavior objects of that type should have.

class Waiter:
    pass

Objects

Individual instances created from the class.

raj = Waiter()
simran = Waiter()

Waiter class→ raj object+ simran object
Class = blueprint. Object = a real instance created from that blueprint.

3. Class Attributes vs Instance Attributes

One of the most common Python OOP interview topics.

Type Defined where? Belongs to Typical use
Class attribute Directly inside the class The class and shared lookup by instances Constants or shared configuration
Instance attribute Usually inside __init__ using self One object Object-specific state 
class Waiter:
    restaurant_name = "Spice House"   # class attribute

    def __init__(self, name):
        self.name = name              # instance attribute
        self.tables = []              # separate list per object

Mutable class attribute trap: A list declared directly in the class is shared unless an instance replaces it. 
class BadWaiter:
    tables = []   # shared mutable object

raj = BadWaiter()
simran = BadWaiter()

raj.tables.append(4)

print(simran.tables)    # [4] — unexpected shared state
Correction: Instance attributes are not “private by default.” They are simply stored on each instance. Python privacy is mostly convention-based, such as _name, with name mangling for __name.
Use class attributes for shared values. Use self.attribute for object-specific data.

4. __init__ and self

How Python initializes and separates object state.

__init__

Runs automatically after a new instance is created. It initializes that instance.

self

Refers to the current instance on which the method is operating.

class Waiter:
    def __init__(self, name, shift):
        self.name = name
        self.shift = shift
        self.tables = []

raj = Waiter("Raj", "Morning")
simran = Waiter("Simran", "Evening")
When Python evaluates raj.take_order(4), it behaves conceptually like Waiter.take_order(raj, 4). The object is passed as the first argument. 
class Waiter:
    def take_order(self, table_number):
        print(f"{self.name} took order from table {table_number}")
self means “this particular object,” not the class as a whole.

5. Methods and Object State

Methods define what an object can do; state is its current data.

class Waiter:
    def __init__(self, name):
        self.name = name
        self.tables = []

    def take_order(self, table_number):
        self.tables.append(table_number)
        return f"{self.name} took table {table_number}'s order."

    def current_tables(self):
        return list(self.tables)

raj = Waiter("Raj")
raj.take_order(4)
raj.take_order(5)

print(raj.current_tables())  # [4, 5]

Instance method

Receives self and works with instance or class data.

Class method

Receives cls. Often used as an alternative constructor.

Static method

Receives neither self nor cls. It is a related utility function.

class Waiter:
    restaurant = "Spice House"

    def __init__(self, name):
        self.name = name

    @classmethod
    def from_badge(cls, badge):
        name = badge.split("-")[0]
        return cls(name)

    @staticmethod
    def is_valid_table(table_number):
        return isinstance(table_number, int) and table_number > 0
State = current object data. Method = behavior that reads or changes that state.

6. Encapsulation

Keep data and the operations that protect it inside one object.

Encapsulation is not just “hiding variables.” It means exposing a controlled public interface and keeping internal rules inside the object.

class Order:
    def __init__(self, table_number):
        self.table_number = table_number
        self._items = []          # internal-use convention
        self.__status = "OPEN"    # name-mangled attribute

    def add_item(self, item):
        if self.__status != "OPEN":
            raise ValueError("Cannot modify a closed order.")
        self._items.append(item)

    def close(self):
        if not self._items:
            raise ValueError("Cannot close an empty order.")
        self.__status = "CLOSED"

    @property
    def total_items(self):
        return len(self._items)
SyntaxMeaning in Python
namePublic by convention
_nameInternal-use convention; still accessible
__nameName mangling helps avoid accidental access/overrides; not true security
@propertyControlled attribute-like access through a method
Encapsulation = protect invariants by making the object responsible for valid state changes.

7. Inheritance

A child class reuses or specializes a parent class.

Waiteris aStaff member 
class Staff:
    def __init__(self, name, shift):
        self.name = name
        self.shift = shift

    def start_work(self):
        return f"{self.name} started the {self.shift} shift."


class Waiter(Staff):
    def __init__(self, name, shift, tables):
        super().__init__(name, shift)
        self.tables = tables

    def take_order(self, table_number):
        return f"{self.name} took an order from table {table_number}."


class Chef(Staff):
    def cook(self, dish):
        return f"{self.name} cooked {dish}."
super(): Delegates to the next class in the method resolution order. In simple single inheritance, it commonly calls the parent implementation.

Good use

There is a genuine “is-a” relationship and the child respects the parent contract.

Bad use

Inheritance is used only to reuse code, creating a rigid and confusing hierarchy.

Inheritance models an is-a relationship.

8. Polymorphism

The same interface can produce different behavior.

class Waiter:
    def __init__(self, name):
        self.name = name

    def work(self):
        return f"{self.name} is taking orders."


class Chef:
    def __init__(self, name):
        self.name = name

    def work(self):
        return f"{self.name} is cooking food."


staff_members = [Waiter("Raj"), Chef("Amit")]

for member in staff_members:
    print(member.work())
Python often uses duck typing: the caller cares that an object supports the required operation, not necessarily that it inherits from a particular class. 
def start_shift(worker):
    print(worker.work())

start_shift(Waiter("Raj"))
start_shift(Chef("Amit"))
Same method call, different behavior based on the object.

9. Abstraction

Expose what an object must do while allowing subclasses to decide how.

An abstract base class can define a contract that concrete subclasses must satisfy.

from abc import ABC, abstractmethod


class Staff(ABC):
    def __init__(self, name):
        self.name = name

    def start_work(self):
        return f"{self.name} started work."

    @abstractmethod
    def work(self):
        # Every concrete staff role must implement this method.
        raise NotImplementedError


class Waiter(Staff):
    def work(self):
        return f"{self.name} is serving customers."


class Chef(Staff):
    def work(self):
        return f"{self.name} is cooking."
# Staff("Generic")  # TypeError: abstract class cannot be instantiated

raj = Waiter("Raj")
print(raj.work())
Important correction: Abstraction is broader than “checking experience.” It reduces unnecessary detail and establishes a stable interface or contract.
Abstraction defines what must exist; concrete classes provide how it works.

10. Composition

Build complex behavior by combining smaller objects.

Waiterhas aChef dependency 
class Chef:
    def __init__(self, name):
        self.name = name

    def cook(self, dish):
        return f"{self.name} cooked {dish}."


class Waiter:
    def __init__(self, name, chef):
        self.name = name
        self.chef = chef

    def take_order(self, dish):
        message = f"{self.name} took the order for {dish}."
        cooking_result = self.chef.cook(dish)
        return f"{message}\n{cooking_result}"


chef = Chef("Amit")
waiter = Waiter("Raj", chef)

print(waiter.take_order("Pasta"))

11. Complete Restaurant Example

All major concepts connected in one clean design.

from abc import ABC, abstractmethod


class Staff(ABC):
    restaurant = "Spice House"

    def __init__(self, name, shift):
        self.name = name
        self.shift = shift

    def start_work(self):
        return f"{self.name} started the {self.shift} shift."

    @abstractmethod
    def work(self):
        raise NotImplementedError


class Chef(Staff):
    def work(self):
        return f"{self.name} is preparing food."

    def cook(self, dish):
        return f"{self.name} cooked {dish}."


class Waiter(Staff):
    def __init__(self, name, shift, chef):
        super().__init__(name, shift)
        self.chef = chef
        self._tables = []

    def work(self):
        return f"{self.name} is serving customers."

    def take_order(self, table_number, dish):
        if table_number <= 0:
            raise ValueError("Table number must be positive.")

        self._tables.append(table_number)
        return (
            f"{self.name} took an order from table {table_number}.\n"
            f"{self.chef.cook(dish)}"
        )

    @property
    def assigned_tables(self):
        return tuple(self._tables)


chef = Chef("Amit", "Morning")
waiter = Waiter("Raj", "Morning", chef)

print(waiter.start_work())
print(waiter.work())
print(waiter.take_order(4, "Pasta"))
print(waiter.assigned_tables)

Concept mapping

  • Class/Object: Waiter and waiter
  • Encapsulation: _tables and controlled updates
  • Inheritance: Waiter(Staff)
  • Polymorphism: role-specific work()
  • Abstraction: abstract Staff.work()
  • Composition: waiter uses a chef

Design benefit

Each class has a focused responsibility. New roles can be added without rewriting the manager's workflow, and the chef dependency can be replaced in tests.

12. Interview Revision

Compact answers for common Python OOP questions.

What are the four pillars of OOP? Encapsulation, inheritance, polymorphism, and abstraction. Composition is not traditionally one of the four pillars, but it is a major OOP design technique.
What is the difference between a class and an object? A class defines structure and behavior. An object is a runtime instance with its own state.
What does self mean? It is the current instance passed to an instance method. The name self is a convention, but strongly recommended.
What is the difference between class and instance attributes? Class attributes are defined on the class and used as shared/default values. Instance attributes are stored separately on each object.
Why is a mutable class attribute dangerous? Every instance may reference the same list, dictionary, or set, causing accidental shared state.
What is method overriding? A child class defines a method with the same name as its parent to provide specialized behavior.
What is method overloading in Python? Python does not support traditional signature-based overloading directly. Use default arguments, *args/**kwargs, singledispatch, or alternative constructors.
What is MRO? Method Resolution Order is the sequence Python follows when searching for methods in an inheritance hierarchy. Inspect it using ClassName.mro().
Inheritance or composition—which is better? Neither universally. Use inheritance for a true substitutable “is-a” relationship; prefer composition for flexible “has-a/uses-a” collaboration.
Is __private truly private in Python? No. Double underscores trigger name mangling, which reduces accidental access but does not provide security.

30-second answer

OOP organizes code into objects that combine state and behavior. A class is a blueprint, and an object is an instance. Encapsulation protects valid state changes, inheritance reuses common behavior, polymorphism lets one interface support different implementations, abstraction defines contracts, and composition connects independent objects for flexible design.

Common mistakes

  • Putting mutable lists or dictionaries directly on the class.
  • Using inheritance only to avoid duplicate code.
  • Creating classes with too many unrelated responsibilities.
  • Using getters and setters without enforcing any real rule.
  • Assuming underscore naming provides real security.
  • Forgetting to call super().__init__() when parent initialization is required.

13. Method Overloading, Overriding & Operator Overloading

Three closely related interview concepts shown side by side.

1. Method Overloading — Python Way

Python does not support traditional signature-based method overloading. If two methods with the same name are defined in one class, the later definition replaces the earlier one. 

class Demo:
    def show(self, value):
        return f"One value: {value}"

    def show(self, first, second):
        return f"Two values: {first}, {second}"


demo = Demo()

# The first show() no longer exists.
print(demo.show(1, 2))
The second method replaces the first method in the class namespace.

Approach A: Default Arguments

class Calculator:
    def add(self, a, b, c=None):
        if c is not None:
            return a + b + c

        return a + b


calc = Calculator()

print(calc.add(2, 3))     # 5
print(calc.add(2, 3, 4))  # 9

Approach B: Variable Arguments

class Multiplier:
    def multiply(self, *args):
        result = 1

        for number in args:
            result *= number

        return result


mult = Multiplier()

print(mult.multiply(2, 3))     # 6
print(mult.multiply(2, 3, 4))  # 24
Python-style overloading uses one method with default arguments, *args, **kwargs, type checks, or singledispatch.

2. Method Overriding

Method overriding occurs when a child class provides its own implementation of a method inherited from a parent class.

Complete Replacement

class Animal:
    def make_sound(self):
        return "Generic animal sound"


class Dog(Animal):
    def make_sound(self):
        return "Bark!"


pet = Dog()

print(pet.make_sound())
# Bark!

Python finds make_sound() in Dog first, so the parent implementation is not used.

Extending with super()

class Employee:
    def get_role(self):
        return "Employee"


class Manager(Employee):
    def get_role(self):
        base_role = super().get_role()
        return f"{base_role} -> Manager"


manager = Manager()

print(manager.get_role())
# Employee -> Manager
super() delegates to the next implementation in the method resolution order rather than hard-coding the parent class name.
Overriding = same method name in parent and child, but the child provides specialized behavior.

3. Operator Overloading

Operator overloading changes how built-in operators behave for custom objects by implementing special dunder methods. 

class Point:
    def __init__(self, x, y):
        self.x = x
        self.y = y

    def __add__(self, other):
        if not isinstance(other, Point):
            return NotImplemented

        return Point(
            self.x + other.x,
            self.y + other.y,
        )

    def __repr__(self):
        return f"Point({self.x}, {self.y})"


p1 = Point(1, 2)
p2 = Point(3, 4)

p3 = p1 + p2

print(p3)  # Point(4, 6)

The expression p1 + p2 internally calls p1.__add__(p2).

Operator Dunder method
+__add__
-__sub__
*__mul__
==__eq__
<__lt__
len()__len__
Operator overloading gives custom objects natural behavior with Python operators.

Overloading vs Overriding vs Operator Overloading

Concept Meaning Python implementation
Method overloading Same conceptual operation accepts different inputs. Default arguments, *args, **kwargs, type checks, or dispatch.
Method overriding A child replaces or extends inherited behavior. Define the same method in the child; optionally call super().
Operator overloading Customizes operators for user-defined objects. Implement dunder methods such as __add__ or __eq__.

Common interview questions

Does Python support method overloading? Not in the traditional compile-time, signature-based sense. Python keeps only the latest method definition with a given name inside the class.
How can overloading-like behavior be implemented? Use default parameters, variable-length arguments, keyword arguments, explicit type checks, functools.singledispatch, or separate clearly named methods.
What is the difference between overriding and overloading? Overriding changes inherited behavior in a subclass. Overloading provides multiple ways to call the same conceptual operation with different inputs.
Why return NotImplemented from __add__? It tells Python that the operation is unsupported for that operand type and allows Python to try the reflected operation on the other object.
Interview memory line: Python has method overriding and operator overloading directly. Traditional method overloading is simulated with flexible function parameters or dispatch techniques.

14. OOP Interview Questions — Product Case Study

Abstract base classes, inheritance, super(), overriding, and expiry-based discount logic.

Likely question: Create an abstract Product class. Create a PerishableProduct subclass that gives a 20% discount when it is close to expiry.

Concepts tested

ConceptHow it appears
Abstract base classProduct inherits from ABC.
Abstract methodcalculate_price() defines the required contract.
InheritanceRegularProduct and PerishableProduct inherit from Product.
super()The child reuses parent initialization and validation.
Method overridingEach child provides its own calculate_price().
PolymorphismThe same method call works differently for each product type.

Complete interview solution

from abc import ABC, abstractmethod
from datetime import date, timedelta


class Product(ABC):
    def __init__(self, name: str, price: float) -> None:
        if not name.strip():
            raise ValueError("Product name cannot be empty")

        if price < 0:
            raise ValueError("Price cannot be negative")

        self.name = name
        self.price = price

    @abstractmethod
    def calculate_price(self) -> float:
        """Return the final selling price."""
        raise NotImplementedError


class RegularProduct(Product):
    def calculate_price(self) -> float:
        return self.price


class PerishableProduct(Product):
    DISCOUNT_RATE = 0.20
    DISCOUNT_WINDOW_DAYS = 3

    def __init__(
        self,
        name: str,
        price: float,
        expiry_date: date,
    ) -> None:
        super().__init__(name, price)
        self.expiry_date = expiry_date

    def calculate_price(self) -> float:
        days_remaining = (self.expiry_date - date.today()).days

        if days_remaining < 0:
            return 0.0

        if days_remaining <= self.DISCOUNT_WINDOW_DAYS:
            return round(
                self.price * (1 - self.DISCOUNT_RATE),
                2,
            )

        return self.price


regular = RegularProduct("Notebook", 100.0)

fresh_milk = PerishableProduct(
    "Milk",
    50.0,
    date.today() + timedelta(days=7),
)

near_expiry_milk = PerishableProduct(
    "Milk",
    50.0,
    date.today() + timedelta(days=2),
)

expired_milk = PerishableProduct(
    "Milk",
    50.0,
    date.today() - timedelta(days=1),
)

print(regular.calculate_price())          # 100.0
print(fresh_milk.calculate_price())       # 50.0
print(near_expiry_milk.calculate_price()) # 40.0
print(expired_milk.calculate_price())     # 0.0
The discount rule is: expired product → 0.0; expiry within three days → 20% discount; otherwise → original price.

Polymorphism example

products: list[Product] = [
    RegularProduct("Notebook", 100.0),
    PerishableProduct(
        "Milk",
        50.0,
        date.today() + timedelta(days=2),
    ),
]

for product in products:
    print(product.name, product.calculate_price())

The loop depends only on the Product contract. It does not need separate conditions for each child class.

Expected follow-up questions

1. Why use ABC? It defines a formal contract. Product represents a general, incomplete concept, while every concrete product must define how its final price is calculated.
2. What happens if the subclass does not implement the abstract method? The subclass also remains abstract, and Python raises TypeError when code tries to instantiate it. 
class BrokenProduct(Product):
    pass


# TypeError: abstract method calculate_price is not implemented
item = BrokenProduct("Invalid", 10.0)
3. Why call super().__init__()? It delegates to the next implementation in the method resolution order. Here it initializes name and price and reuses the validation defined in Product.
4. Abstract method versus concrete method? An abstract method specifies behavior that subclasses must implement. A concrete method already contains usable behavior and is inherited unless the child overrides it. 
class Product(ABC):
    def display_name(self) -> str:       # concrete
        return self.name

    @abstractmethod
    def calculate_price(self) -> float:  # abstract
        raise NotImplementedError
5. Overriding versus overloading? Overriding means a child replaces an inherited method with its own implementation. Python does not provide traditional signature-based overloading like Java. Similar behavior is implemented with default arguments, *args, **kwargs, or functools.singledispatch.

9. How would you add another discount without changing the class? Use composition with the Strategy pattern. Move discount calculation into separate strategy objects and inject the required strategy into the product.

Extensible version using the Strategy pattern

from abc import ABC, abstractmethod
from datetime import date


class DiscountStrategy(ABC):
    @abstractmethod
    def apply(
        self,
        price: float,
        expiry_date: date | None = None,
    ) -> float:
        raise NotImplementedError


class NoDiscount(DiscountStrategy):
    def apply(
        self,
        price: float,
        expiry_date: date | None = None,
    ) -> float:
        return price


class NearExpiryDiscount(DiscountStrategy):
    def __init__(
        self,
        days: int = 3,
        rate: float = 0.20,
    ) -> None:
        self.days = days
        self.rate = rate

    def apply(
        self,
        price: float,
        expiry_date: date | None = None,
    ) -> float:
        if expiry_date is None:
            return price

        days_remaining = (expiry_date - date.today()).days

        if days_remaining < 0:
            return 0.0

        if days_remaining <= self.days:
            return round(price * (1 - self.rate), 2)

        return price


class Product:
    def __init__(
        self,
        name: str,
        price: float,
        discount_strategy: DiscountStrategy,
        expiry_date: date | None = None,
    ) -> None:
        if price < 0:
            raise ValueError("Price cannot be negative")

        self.name = name
        self.price = price
        self.expiry_date = expiry_date
        self.discount_strategy = discount_strategy

    def calculate_price(self) -> float:
        return self.discount_strategy.apply(
            self.price,
            self.expiry_date,
        )


milk = Product(
    name="Milk",
    price=50.0,
    expiry_date=date.today(),
    discount_strategy=NearExpiryDiscount(),
)

notebook = Product(
    name="Notebook",
    price=100.0,
    discount_strategy=NoDiscount(),
)
New festive, loyalty, or clearance discounts can be added as new strategy classes without modifying Product. This supports the Open/Closed Principle.

More OOP interview questions

10. Can an abstract class have a constructor and concrete methods? Yes. An abstract class may contain __init__, attributes, properties, concrete methods, class methods, static methods, and abstract methods.
11. Can an abstract method contain code? Yes. It may contain reusable default behavior, but the subclass must still override it before becoming concrete. The child can call the parent's implementation with super().
12. Can an abstract class be instantiated? Not while it has unimplemented abstract methods.
13. What is MRO? Method Resolution Order is the sequence Python follows to find methods in an inheritance hierarchy. Inspect it with ClassName.mro().
14. What is the Open/Closed Principle? A class should be open for extension but closed for modification. The strategy design follows it because new discounts are added through new classes rather than by editing existing product code.
15. Should an expired product really return 0.0? It is a simplified business rule for this interview problem. In production, expired stock would normally be marked unsellable or produce a domain-specific validation error.

30-second interview answer

Product is an abstract base class that stores common state and forces every concrete product to implement calculate_price(). The child class calls super().__init__() to reuse parent initialization and validation, then overrides the abstract method with expiry-aware pricing. For many changing discount rules, I would use composition and inject a discount strategy so new discounts can be introduced without modifying the product class.
