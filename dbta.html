dbt Mental Model

Transformation as version-controlled software executed inside your data platform.
Raw data→Sources→Staging→Intermediate→Marts→Semantic Layer / BI / AI
Compile

SQL + Jinja

dbt parses the project, resolves dependencies, renders templates, and produces warehouse-specific SQL.

Execute

Pushdown ELT

Compute happens in Snowflake, Databricks, BigQuery, Redshift, Postgres, or another adapter-supported platform.

Govern

Metadata graph

Models, tests, owners, contracts, versions, exposures, and metrics become one lineage-aware graph.

What dbt does and does not do

dbt does

  • Transform, test, document, and organize analytical data.
  • Compile dependencies through ref() and source().
  • Apply repeatable development, CI, deployment, and governance practices.
  • Define governed metrics through the Semantic Layer and MetricFlow.

dbt usually does not

  • Extract from operational systems or move raw data between platforms.
  • Replace a general workflow orchestrator for all non-dbt tasks.
  • Automatically guarantee good modeling, grain, or metric definitions.
  • Fix poor warehouse design or inefficient SQL without engineering decisions.
Interview answer“dbt brings software-engineering practices to analytics transformations: modular code, dependency management, tests, documentation, CI/CD, lineage, and governed metrics—all executed where the data already lives.”

Project Anatomy and Configuration

Know what belongs in the repository, what belongs in credentials, and how configs cascade.

Common repository structure

  • dbt_project.yml — project paths and defaults
  • models/ — SQL/Python models and YAML properties
  • macros/ — reusable Jinja
  • tests/ — singular data tests
  • seeds/, snapshots/, analyses/
  • packages.yml or dependencies.yml
  • selectors.yml — named selection definitions

Outside or environment-specific

  • profiles.yml for local Core credentials and targets
  • dbt platform environment connections and deployment jobs
  • Secrets through environment variables or secret managers
  • target/, logs/, dbt_packages/ generated locally

Configuration precedence

Most specific normally winsProject defaults → directory-level config → resource YAML config → model-level config() call. Command flags can override runtime behavior such as --full-refresh.

var()

Project/runtime variables for non-secret behavior. Keep stable business logic out of ad-hoc CLI variables.

env_var()

Environment-specific values and secrets. Never commit credentials into the repository.

target

Runtime context such as target name, schema, database, and adapter type.

DAG, Dependencies, ref(), and source()

The DAG is dbt's execution and lineage backbone.
models/marts/fct_orders.sql
with orders as (
    select *
    from {{ source('raw_commerce', 'orders') }}
),
customers as (
    select *
    from {{ ref('stg_customers') }}
)
select
    o.order_id,
    o.customer_id,
    c.customer_segment,
    o.ordered_at,
    o.order_total
from orders o
left join customers c using (customer_id)

ref()

  • Creates dependency and lineage.
  • Resolves environment/database/schema naming.
  • Supports versioned refs.
  • Enables selection, defer, and impact analysis.

source()

  • Declares an external/raw dependency.
  • Centralizes physical relation mapping.
  • Enables source tests and freshness.
  • Separates ingestion ownership from transformation ownership.

Modeling Layers and Naming

Separate source cleanup, reusable transformations, and business-facing models.
Staging

stg_*

One source-aligned model per raw object. Rename, cast, standardize, deduplicate, and expose a clean source interface.

Intermediate

int_*

Reusable joins, pivots, event preparation, and business transformations that are not yet consumer-facing.

Marts

fct_*, dim_*

Stable, documented, tested data products aligned to clear grain and business processes.

Grain-first modeling

  1. State the grain in one sentence: “one row per order,” “one row per customer per day.”
  2. Identify primary/business keys and expected uniqueness.
  3. Aggregate lower-grain children before joining to avoid fan-out.
  4. Choose additive, semi-additive, and non-additive measures deliberately.
  5. Document filters, currency, timezone, status, cancellation, and late-data rules.
Strong design principleA model should have one clear purpose, one declared grain, explicit columns, deterministic logic, and a test strategy tied to business risk.

Choose based on cost, freshness, debugging, reuse, and platform behavior—not habit.

    materialized='table',
    schema='analytics',
    alias='customer_360',

Decision framework

Prefer physical materialization when

The model is expensive, reused, latency-sensitive, contract-governed, or a stable interface.

Prefer logical/inlined materialization when

The logic is lightweight, changes frequently, is used by few consumers, or does not justify storage.

Incremental Models Deep Dive

The most important advanced dbt interview topic: performance is easy; correctness is hard.

Core pattern

Incremental merge with lookback and deduplication
{{ config(
    materialized='incremental',
    incremental_strategy='merge',
    unique_key='order_id',
    on_schema_change='sync_all_columns'
) }}

with source_rows as (
    select *
    from {{ ref('stg_orders') }}

    {% if is_incremental() %}
      -- Example uses warehouse-specific timestamp arithmetic.
      where updated_at >= (
        select dateadd(day, -3, max(updated_at)) from {{ this }}
      )
    {% endif %}
),
deduplicated as (
    select *
    from source_rows
    qualify row_number() over (
      partition by order_id
      order by updated_at desc, ingestion_sequence desc
    ) = 1
)
select * from deduplicated

Watermark

The event/update timestamp or sequence used to reduce the source scan.

Lookback

Reprocess recent periods to catch late or corrected records.

Unique key

Stable key used to update/replace matching target rows.

Strategies

StrategyBehaviorUseRisk
appendInsert incoming rows onlyImmutable events with guaranteed no duplicatesCorrections and retries create duplicates
mergeMatch by key, update and insertMutable records and CDC-like feedsKey duplication, expensive target scan
delete+insertDelete matching keys, then insertAdapters/workloads where merge is unsuitableTransactional and performance considerations
insert_overwriteReplace partitionsPartitioned facts and batch correctionsWrong partition predicate overwrites data
microbatchProcess time windows as batchesVery large event streamsRequires event-time discipline and batch planning

Microbatch example

Time-based microbatch configuration
{{ config(
    materialized='incremental',
    incremental_strategy='microbatch',
    event_time='event_ts',
    begin='2024-01-01',
    batch_size='day',
    lookback=2
) }}

select
    event_id,
    user_id,
    event_ts,
    event_type
from {{ ref('stg_events') }}

Correctness checklist

High-value interview statementAn incremental model is not correct merely because it runs faster. You must prove that retries, late data, updates, deletes, schema changes, and backfills converge to the intended target state.

Sources, Seeds, Snapshots, and Freshness

External inputs, small reference data, history capture, and recency SLAs.
Source declaration and freshness
version: 2

sources:
  - name: raw_commerce
    database: "{{ env_var('RAW_DATABASE') }}"
    schema: commerce
    loader: fivetran
    loaded_at_field: _loaded_at
    freshness:
      warn_after: {count: 2, period: hour}
      error_after: {count: 6, period: hour}
    tables:
      - name: orders
        description: Raw order feed
        columns:
          - name: order_id
            data_tests: [not_null]
          - name: status
            data_tests:
              - accepted_values:
                  arguments:
                    values: [placed, shipped, completed, cancelled]

Sources

Declared external relations with lineage, tests, metadata, and freshness.

Seeds

Small, static CSV mappings such as country codes or manual categories.

Snapshots

Historical versions of mutable source records, typically SCD Type II.

{{ config(
    target_schema='snapshots',

Testing, Contracts, and Data Quality

Tests should encode risks, not merely inflate coverage numbers.

Generic data tests and contract

Properties YAML
version: 2

models:
  - name: fct_orders
    description: One row per order
    config:
      contract:
        enforced: true
    columns:
      - name: order_id
        data_type: bigint
        data_tests: [not_null, unique]
      - name: customer_id
        data_type: bigint
        data_tests:
          - relationships:
              arguments:
                to: ref('dim_customers')
                field: customer_id
      - name: order_status
        data_type: varchar
        data_tests:
          - accepted_values:
              arguments:
                values: [placed, shipped, completed, cancelled]
              config:
                severity: warn
                warn_if: "> 0"
                error_if: "> 100"

Singular data test

Singular test
-- tests/assert_order_total_non_negative.sql
-- Return failing rows.
select *
from {{ ref('fct_orders') }}
where order_total < 0
  and order_status <> 'refund'

Unit test

Controlled model logic test
unit_tests:
  - name: order_total_excludes_cancelled_items
    model: int_order_totals
    given:
      - input: ref('stg_order_items')
        rows:
          - {order_id: 1, item_amount: 100, item_status: shipped}
          - {order_id: 1, item_amount: 25, item_status: cancelled}
    expect:
      rows:
        - {order_id: 1, order_total: 100}

Data tests

Validate built data and business assertions. Failures are rows returned by a query.

Unit tests

Validate model logic with fixtures before materialization during dbt build.

Contracts

Enforce output columns and types as an interface guarantee.

Quality layers

  1. Schema: not-null, unique, accepted values, types.
  2. Relationships: orphan keys and referential integrity.
  3. Business rules: balances, status transitions, date ordering.
  4. Reconciliation: source-to-target counts, sums, and control totals.
  5. Statistical: volume, distribution, drift, anomalies.
  6. Operational: freshness, runtime, failures, retries, SLA.

Jinja, Macros, Hooks, and Dispatch

Use abstraction to remove repetition—not to hide the data model.
Reusable macro
{% macro cents_to_currency(column_name, scale=2) %}
    round({{ column_name }} / 100.0, {{ scale }})
{% endmacro %}

select
    payment_id,
    {{ cents_to_currency('amount_cents') }} as amount
from {{ ref('stg_payments') }}
Parse-safe run_query pattern
{% macro get_payment_methods() %}
  {% set sql %}
    select distinct payment_method
    from {{ ref('stg_payments') }}
    order by 1
  {% endset %}

  {% if execute %}
    {% set result = run_query(sql) %}
    {{ return(result.columns[0].values()) }}
  {% else %}
    {{ return([]) }}
  {% endif %}
{% endmacro %}

Good macro uses

  • Cross-database functions and repeated SQL patterns
  • Reusable generic tests
  • Metadata-driven column generation
  • Audit operations and controlled maintenance
  • Adapter dispatch for portability

Macro smells

  • Business logic impossible to read in compiled SQL
  • Warehouse queries during parse without execute
  • Dynamic dependencies hidden from the parser
  • Excessive loops generating hundreds of columns
  • Hooks duplicating native grants/config functionality

CI/CD and Environment Strategy

Separate fast developer feedback from production confidence.
Development

Isolated schemas

Per-developer targets, small selections, defer/clone, sample or limited data, fast lint/parse/unit tests.

Pull Request

Slim CI

Parse, static validation, build changed graph, unit/data tests, contract checks, documentation and lineage impact.

Production

Scheduled/triggered jobs

Freshness gates, controlled builds, retries, notifications, artifacts, observability, and rollback/backfill runbooks.

Recommended pipeline

  1. Format/lint SQL and YAML.
  2. Run dbt deps with locked versions.
  3. Run dbt parse and deprecation checks.
  4. Run unit tests for changed models.
  5. Build changed nodes with state/defer.
  6. Run critical data and reconciliation tests.
  7. Review changed public contracts, versions, exposures, and metrics.
  8. Persist manifest/run results for deployment and observability.
Deployment answerPromote code, not development tables. Production should rebuild or safely clone/materialize resources under production credentials and governance controls.

  - name: finance
    owner:
      name: Finance Analytics

    latest_version: 2

      - name: revenue

Breaking versus non-breaking changes

Potentially breaking

  • Remove/rename a column
  • Change grain or key semantics
  • Change type/precision incompatibly
  • Change metric business definition
  • Tighten filters or history unexpectedly

Usually safer

  • Add a nullable column
  • Improve documentation
  • Add tests without blocking historical anomalies
  • Optimize SQL with identical output
  • Add a new model/metric version

Performance and Cost Optimization

Optimize warehouse work, graph scope, and developer feedback loops.

SQL

  • Filter early and prune columns
  • Aggregate before grain-expanding joins
  • Avoid repeated expensive CTE/view chains
  • Use explain/query profile

Materialization

  • Incremental large facts
  • Physicalize reused intermediates
  • Use partitions/clustering/optimization
  • Choose merge predicates carefully

Execution

  • Right-size threads and warehouse
  • Use selectors and state
  • Run independent models concurrently
  • Separate heavy backfills

Slow model investigation

  1. Measure model execution and upstream timing from artifacts.
  2. Inspect compiled SQL and query profile.
  3. Check bytes/partitions scanned and join cardinality.
  4. Determine whether repeated logic should be materialized.
  5. Verify incremental filters reduce the source and target scans.
  6. Re-test correctness after optimization.
Best answer“I optimize the costliest query path, not the prettiest dbt file. Compiled SQL and warehouse query profiles are the source of truth.”

Python Models and User-Defined Functions

Know the boundary between SQL transformations, Python execution, macros, and warehouse UDFs.

Python model

Returns a dataframe using a supported adapter/runtime. Useful for logic awkward in SQL, but less portable and often harder to review.

Macro

Generates SQL at compile time. It is not a warehouse function and cannot be directly called by arbitrary external queries.

UDF resource

A function registered in the data platform and reusable by dbt and external tools. Supported as a first-class resource in newer dbt versions/adapters.

Conceptual Python model
def model(dbt, session):
    dbt.config(materialized="table")
    orders = dbt.ref("stg_orders")

    # API differs by supported dataframe runtime/adapter.
    return orders.filter(orders["order_total"] >= 0)
Interview balancePrefer SQL for transparent set-based transformations. Use Python when libraries, complex procedural logic, or supported ML/dataframe operations provide a clear advantage.

MetricFlow and the Semantic Layer

Central metric definitions, valid dimensions, automatic joins, and generated SQL.
dbt models→semantic annotations→semantic graph→MetricFlow query plan→warehouse SQL→BI / API / AI

Semantic models

Connect physical dbt models to business entities, dimensions, time, and simple metrics.

Metrics

Govern quantitative definitions and composition.

Query engine

Find valid dimensions and join paths, generate optimized platform SQL, and return metric datasets.

Why it matters

  • One definition of revenue instead of one per dashboard.
  • Metric logic is version-controlled, reviewed, documented, and lineage-aware.
  • Consumers request metrics and dimensions rather than writing joins.
  • Entity semantics help prevent invalid fan-out and chasm joins.
  • Saved queries provide governed, reusable consumption patterns.
Interview answerMetricFlow is not just a metric registry. It builds a semantic graph and generates SQL for valid metric-and-dimension combinations, including automatic join planning and advanced metric types.

Latest Semantic Layer YAML Specification

Supported in dbt platform Latest, Fusion, and dbt Core v1.12 as of July 2026.
Major migration changesSemantic configuration is embedded under models; entities and dimensions attach to columns; simple metrics replace measures; type_params is deprecated and its properties move to direct metric keys.
Current embedded model + metrics syntax
version: 2

models:
  - name: fct_orders
    description: One row per order

    semantic_model:
      enabled: true
      name: orders

    agg_time_dimension: ordered_at

    columns:
      - name: order_id
        entity:
          name: order
          type: primary

      - name: customer_id
        entity:
          name: customer
          type: foreign

      - name: ordered_at
        granularity: day
        dimension:
          type: time

      - name: order_status
        dimension:
          type: categorical

      - name: country
        dimension:
          type: categorical

    metrics:
      - name: revenue
        label: Revenue
        type: simple
        agg: sum
        expr: order_total
        fill_nulls_with: 0

      - name: orders
        label: Orders
        type: simple
        agg: count_distinct
        expr: order_id

metrics:
  - name: average_order_value
    label: Average order value
    type: ratio
    numerator: revenue
    denominator: orders

  - name: revenue_30d
    type: cumulative
    input_metric: revenue
    window: 30d
    period_agg: sum

  - name: revenue_growth_7d
    type: derived
    expr: current_revenue - prior_revenue
    input_metrics:
      - name: revenue
        alias: current_revenue
      - name: revenue
        alias: prior_revenue
        offset_window: 7 days

Legacy pattern you will still see

Legacy top-level semantic model and measure
semantic_models:
  - name: orders
    model: ref('fct_orders')
    defaults:
      agg_time_dimension: ordered_at
    entities:
      - name: order
        type: primary
        expr: order_id
    dimensions:
      - name: ordered_at
        type: time
        type_params:
          time_granularity: day
    measures:
      - name: revenue
        agg: sum
        expr: order_total

metrics:
  - name: revenue
    type: simple
    type_params:
      measure: revenue

Lead with latest spec

It is simpler and aligns semantic annotations directly with physical model columns.

Recognize legacy spec

It remains important in existing projects, package compatibility, and some cross-project Semantic Layer patterns.

Metric Types

Choose the semantic abstraction that matches the business calculation.
Metric typeMeaningExamples
SimpleDirect aggregation over an expressionRevenue, orders, active users
RatioNumerator divided by denominatorConversion rate, average order frequency
DerivedExpression over input metricsGross profit, growth, variance
CumulativeAccumulation over all time/window/grain-to-dateMTD revenue, 7-day active users
ConversionBase event followed by conversion event for an entitySignup-to-paid conversion

Examples

Metric type patterns
# Simple metric inside a model
metrics:
  - name: customers
    type: simple
    agg: count_distinct
    expr: customer_id

# Ratio metric at top level
metrics:
  - name: conversion_rate
    type: ratio
    numerator: completed_orders
    denominator: sessions

# Derived metric with offset
  - name: revenue_growth_rate_7d
    type: derived
    expr: (current_revenue - prior_revenue) / nullif(prior_revenue, 0)
    input_metrics:
      - {name: revenue, alias: current_revenue}
      - {name: revenue, alias: prior_revenue, offset_window: 7 days}

# Cumulative metric
  - name: revenue_mtd
    type: cumulative
    input_metric: revenue
    grain_to_date: month
    period_agg: sum

# Conversion metric
  - name: signup_to_paid
    type: conversion
    entity: user
    calculation: conversion_rate
    base_metric: signups
    conversion_metric: paid_signups
    window: 30 days
Metric design ruleDo not encode an ambiguous business definition merely because the YAML validates. Document eligibility, grain, time basis, filters, currency, status, late updates, and ownership.

Entities, Join Logic, Grain, and Time

Most semantic-layer errors are really grain and cardinality errors.
Entity typeMeaningTypical example
PrimaryOne row per entity at the semantic model grainorder in one-row-per-order fact
UniqueUnique in this model but not necessarily complete across the platformactive-customer subset
ForeignMay repeat and points to another entitycustomer_id on orders
NaturalReal-world identifying key, possibly composedsalesperson + department assignment

Safe semantic graph

orders (primary order, foreign customer)→customers (primary customer)→region dimension

Fan-out example

Wrong grainJoining one-row-per-order revenue directly to one-row-per-item data duplicates order revenue. Aggregate item data to order grain first or model a separate item metric at item grain.

Time dimension

Declare native granularity and default agg_time_dimension. Be explicit about event time versus processing time.

Time spine

Provides continuous periods for cumulative metrics, filling missing values, and consistent time-grain alignment.

SCD Type II dimensions

MetricFlow can support time-aware joins to SCD Type II dimensions when the semantic model is configured correctly. The business question must specify whether attributes are evaluated “as of event time” or using the latest dimension state.

Building a Trusted Layer for Humans and AI Agents

A semantic layer is trusted only when the underlying data product is governed and observable.
Truth

Definition

Metric formula, grain, filters, time basis, currency, exclusions, owner, and certification status.

Safety

Join semantics

Entities, cardinality, fan-out controls, dimensional reachability, and valid query combinations.

Reliability

Operations

Freshness, tests, reconciliation, incidents, SLAs, artifacts, and usage monitoring.

Implementation sequence

  1. Choose a high-value domain and identify canonical business processes.
  2. Build grain-correct facts/dimensions with stable keys.
  3. Enforce contracts, data tests, and source-to-finance reconciliations.
  4. Assign groups, owners, access, and versioning policy.
  5. Define semantic entities, dimensions, time, and simple metrics.
  6. Compose ratios, derived, cumulative, and conversion metrics.
  7. Validate allowed dimensions and generated SQL.
  8. Publish saved queries for executive and AI consumption.
  9. Monitor freshness, quality, latency, cost, and metric usage.

AI agent guardrails

  • Allow queries only against certified metrics and dimensions.
  • Return metric definition, filters, grain, and freshness with every answer.
  • Limit expensive dimensions/time ranges and enforce row-level policies.
  • Log prompts, resolved semantic queries, generated SQL, runtime, and feedback.
  • Use deterministic metric APIs instead of asking an LLM to invent SQL joins.

Scenario-Based Interview Questions

Answer with diagnosis, correctness, trade-offs, and prevention.
A daily incremental model misses records that arrive two days late. What do you change?
Use a lookback window on the source filter, merge by a stable unique key, keep the predicate on the source scan, and add a reconciliation test comparing recent source and target counts or sums.
A merge model suddenly creates duplicate customer rows.
Verify the incoming dataset has one row per unique_key, deduplicate with row_number before the merge, test key uniqueness in staging, and confirm the business key has not changed semantics.
CI builds every model and costs too much.
Persist the production manifest, select state:modified plus required descendants, defer unchanged parents to production, exclude nonessential heavy models, and use schema-only or sampled validation where appropriate.
A downstream dashboard broke after a column rename.
Introduce a new model version or compatibility alias, enforce contracts, mark the old version with a deprecation date, notify exposure owners, and migrate consumers before removal.
Two teams calculate revenue differently.
Agree on grain, inclusion/exclusion rules, time basis, currency handling, refunds, and late data; implement one governed metric in the semantic layer; reconcile it to finance controls; publish ownership and certification.
MetricFlow refuses a requested dimension.
Check whether the dimension is reachable through valid entity paths, whether entity names/types align, and whether the requested join would create fan-out or chasm behavior. Remodel the grain instead of forcing an unsafe join.
Revenue doubles after joining orders to order items.
The join changed grain. Aggregate items to order grain before joining, or define semantic models/entities so MetricFlow avoids fan-out. Add reconciliation tests on totals before and after the join.
A source freshness check is green but data is wrong.
Freshness measures recency, not completeness or correctness. Add volume, null, distribution, accepted-value, and reconciliation checks plus ingestion metadata monitoring.
A macro works in run but fails during parse.
It likely accesses query results during parse. Wrap runtime-only logic in if execute and provide a parse-safe default branch.
An ephemeral chain produces unreadable compiled SQL.
Materialize stable intermediate boundaries as views or tables, reduce abstraction depth, and reserve ephemeral models for small reusable transformations.
A full refresh takes six hours.
Confirm whether it is truly required; use incremental or microbatch processing, partition pruning, staged rebuilds, warehouse scaling, and controlled backfill windows. Validate results before swapping.
A contract blocks a harmless numeric change.
Treat it as an interface change. Check adapter type aliasing and precision behavior, update the contract deliberately, test downstream compatibility, and version the model if the change is breaking.

Rapid-Fire Interview Q&A

Use the search box to filter this bank by concept.
What problem does dbt solve?
dbt transforms data already loaded in a warehouse or lakehouse. It adds modular SQL/Python models, dependency management, testing, documentation, lineage, reusable macros, environments, and software-engineering workflows around transformations.
Is dbt an ETL tool?
Primarily it is the T in ELT. Extraction and loading are usually handled by ingestion tools, while dbt compiles and executes transformations inside the data platform.
What is a dbt model?
A select statement, or supported Python model, that dbt materializes as a relation or inlines as an ephemeral CTE. Its filename is the default model name.
What does ref() do?
It resolves the correct database relation, creates a DAG dependency, supports environment-aware naming, and enables lineage and selection. Hard-coded relation names lose these benefits.
What does source() do?
It references declared raw inputs, creates source lineage, centralizes database/schema naming, and enables source tests and freshness checks.
dbt run versus dbt build?
dbt run executes models. dbt build executes selected models, tests, seeds, snapshots, and supported functions in DAG order; failing upstream tests can skip downstream nodes.
compile versus run?
compile renders Jinja and writes compiled SQL without materializing models. run compiles and executes model SQL in the target platform.
Why use staging models?
They isolate source-specific cleanup: renaming, casting, basic standardization, deduplication, and source-aligned tests. Business logic should normally live downstream.
What is an ephemeral model?
A model inlined as a CTE into downstream compiled SQL. It creates no warehouse relation, which can reduce clutter but make debugging and deeply nested SQL harder.
When should a model be a table?
When queries are expensive, reused often, require predictable performance, or form a stable interface for downstream consumers.
When should a model be a view?
For lightweight logic, low storage overhead, rapid iteration, and when warehouse optimization can efficiently execute the underlying query.
What is an incremental model?
A model that processes only new or changed records after the initial build. Correctness depends on a reliable filter, merge strategy, key, late-data handling, and idempotent logic.
What does is_incremental() check?
It is true when the target relation exists, the model is configured incremental, and the run is not a full refresh. Use it to conditionally restrict source rows.
Why is unique_key important?
For update-capable strategies such as merge, it identifies matching target rows. A non-unique or unstable key can create duplicates, merge errors, or incorrect updates.
How do you handle late-arriving data?
Reprocess a lookback window, merge by a stable key, and monitor lateness. The lookback increases cost slightly but protects correctness when timestamps arrive late.
What is microbatch incremental processing?
Time-based incremental execution split into batches using an event-time column. It improves manageability and retry behavior for very large event datasets.
What is on_schema_change?
It controls how incremental models react when source/model columns change, such as ignoring changes, failing, appending new columns, or synchronizing columns where supported.
Snapshot versus incremental model?
A snapshot preserves historical row versions, usually for SCD Type II behavior. An incremental model efficiently builds the current or transformed dataset and does not automatically retain history.
Timestamp versus check snapshot strategy?
Timestamp tracks changes using a reliable updated-at column and is generally preferred. Check compares selected columns and is useful when no trustworthy update timestamp exists.
Data test versus unit test?
A data test queries built data and returns failing rows. A unit test supplies controlled input fixtures and validates a model's SQL logic before materializing that model in dbt build.
What are the four common generic tests?
unique, not_null, accepted_values, and relationships. They cover basic key, completeness, domain, and referential-integrity expectations.
Test versus contract?
Tests validate data values after or around execution. Contracts enforce the declared output shape and data types during model creation, making the model interface explicit.
How do warning thresholds work?
Configure severity plus warn_if/error_if expressions. This allows soft monitoring for small failures and hard blocking when a threshold is exceeded.
What is store_failures?
It persists failing rows to an audit relation so engineers can inspect, trend, and remediate bad records.
What is a macro?
A reusable Jinja function that generates SQL or orchestrates metadata operations. Macros reduce duplication but should be used carefully to avoid hiding business logic.
Why use adapter.dispatch?
It lets a macro provide adapter-specific implementations behind one public interface, enabling portable packages and warehouse-optimized SQL.
What is run_query()?
A Jinja function that executes SQL during compilation/execution context and returns a result object. Guard result-dependent logic with execute to avoid parse-time errors.
Why use if execute?
Parsing builds the manifest without necessarily querying the warehouse. if execute prevents code from accessing runtime query results during parse-only phases.
What are pre-hooks and post-hooks?
SQL statements run before or after a resource. Typical uses include grants, audit logging, and maintenance; native configs are preferred when available.
What is a model contract?
An enforced declaration of model columns and types. It helps protect downstream users from unreviewed breaking schema changes.
What are model versions?
Named versions of a model interface that let producers introduce breaking changes while keeping older consumers operational during migration.
What are model access levels?
Private restricts references to the same group, protected allows references within the project, and public is intended as a cross-project interface where supported.
What are groups?
Ownership boundaries for dbt resources. Groups support governance, access control, and clearer accountability in large projects.
What is an exposure?
Metadata describing a downstream dashboard, application, notebook, ML workflow, or other consumer. It extends lineage beyond dbt models.
What is dbt Mesh?
A multi-project operating model using governed public interfaces, ownership, contracts, versions, and cross-project dependencies to scale domain-oriented development.
What is state:modified?
A selector that compares the current project to a prior manifest and selects changed resources. Subselectors can distinguish body, configs, dependencies, macros, and other changes.
What is defer?
It resolves unbuilt references to relations represented in a supplied state manifest, commonly production, so developers can test changed models without rebuilding every upstream dependency.
What is Slim CI?
A CI pattern that builds changed nodes and relevant descendants while deferring unchanged upstream relations to a production-like state.
Why use --favor-state?
It prefers deferred state relations even when a relation exists in the current target, reducing accidental use of stale or partial developer objects.
What are dbt artifacts?
Machine-readable JSON outputs such as manifest, run_results, catalog, sources freshness results, and semantic_manifest. They power lineage, observability, impact analysis, and custom tooling.
manifest.json versus run_results.json?
manifest describes project resources, configs, dependencies, and compiled metadata. run_results records execution status, timing, adapter responses, and failures for a specific invocation.
How do you optimize a slow dbt project?
Measure first; reduce full scans, use appropriate materializations, filter early, prune columns, optimize incremental predicates, control threads, improve warehouse physical design, and use state-aware selection.
How do you debug a model failure?
Read the first database error, inspect target/compiled SQL, run the smallest failing selection, verify credentials and relation names, check upstream tests/freshness, and compare environment-specific configs.
Why avoid SELECT * in marts?
It creates accidental interface changes, scans unnecessary columns, weakens contracts and documentation, and makes downstream breakage harder to detect.
What is a seed?
A version-controlled CSV loaded by dbt, suitable for small static mappings and reference data—not large or frequently changing datasets.
What is source freshness?
A check that compares the newest load timestamp or warehouse metadata with configured warn/error service levels.
How do packages work?
packages.yml or dependencies.yml declares reusable dbt packages. dbt deps resolves them, and a lock file can pin the installed dependency graph.
What is MetricFlow?
A SQL query-generation engine that uses semantic models and metric definitions to create consistent metric queries across dimensions and valid join paths.
What is the dbt Semantic Layer?
A governed serving layer where metrics are centrally defined in a dbt project and queried by BI, notebooks, APIs, and applications without reimplementing metric logic.
What is a semantic model?
The semantic annotation of a dbt model: entities, dimensions, time behavior, and simple metrics. MetricFlow uses these definitions to build a semantic graph.
Entity versus dimension?
An entity identifies a business object and provides join semantics. A dimension is an attribute used to group or filter metric results.
Primary, unique, foreign, and natural entities?
Primary identifies the row grain; unique is unique within a subset; foreign can repeat and joins outward; natural represents a real-world identifying key, including composite expressions where supported.
What is agg_time_dimension?
The default time dimension used to aggregate metrics from a semantic model. Individual metrics can override it.
What are simple metrics in the latest spec?
Aggregations defined directly inside the model's metrics block using keys such as agg and expr. They replace legacy measures.
What are derived metrics?
Metrics calculated from one or more input metrics using an expression, including period offsets for growth and retention calculations.
What are ratio metrics?
A numerator metric divided by a denominator metric, with optional filtering and null-handling behavior.
What are cumulative metrics?
Metrics accumulated over all time, a sliding window, or grain-to-date period using an input metric and time spine.
What are conversion metrics?
Metrics that relate a base event to a later conversion event for the same entity within a configured window and optional constant properties.
How does MetricFlow choose joins?
It builds a graph from entity definitions, selects valid paths, commonly uses left joins from facts to dimensions and full outer joins across multiple facts, and blocks paths likely to create fan-out or chasm errors.
What is a time spine?
A continuous date/timestamp relation used to fill missing periods, align grains, and compute cumulative or null-filled metrics consistently.
What is a saved query?
A reusable, governed metric query definition containing metrics, group-bys, filters, ordering, limits, and optional exports.
How do you validate MetricFlow configs?
Run dbt parse to refresh semantic_manifest.json, then use dbt sl validate in connected platform workflows or mf validate-configs for local open-source MetricFlow workflows.
How do you query a metric from the CLI?
Use dbt sl query --metrics <name> --group-by <dimension> in dbt platform-connected workflows, or mf query with equivalent options in local MetricFlow.
What changed in the 2026 Semantic Layer YAML spec?
Semantic definitions move under models, entities/dimensions attach to columns, simple metrics replace measures, and type-specific properties move out of the deprecated type_params wrapper.
Should you memorize legacy MetricFlow YAML?
Know how to recognize it because many production projects still use it, but lead with the latest supported spec and explain the migration differences.
How would you build a trusted executive metric layer?
Define certified grain-aligned marts, enforce contracts and tests, assign owners, document metric semantics, model entities and dimensions, prevent fan-out joins, validate with reconciliation tests, expose saved queries, and monitor freshness and usage.

End-to-End Design Exercise

Design an executive subscription scorecard with governed metrics.

Prompt

You receive raw customers, subscriptions, invoices, payments, and product events. Build trusted metrics for monthly recurring revenue, new MRR, expansion, contraction, churned MRR, logo churn, active customers, and net revenue retention.

Strong solution outline

  1. Sources: declare raw tables, freshness, ingestion metadata, and primary-key tests.
  2. Staging: normalize timestamps/currency/status; deduplicate CDC records.
  3. Intermediate: create subscription state intervals and invoice/payment reconciliation.
  4. Marts: one row per customer-subscription-day or month for movement logic; separate current customer dimension.
  5. Incremental: microbatch or partition overwrite by event/billing date with a late-data lookback.
  6. Quality: movement sums reconcile beginning MRR to ending MRR; paid invoice totals reconcile finance controls.
  7. Governance: finance owner, public contracted models, versions for definition changes, executive dashboard exposure.
  8. Semantic Layer: customer/subscription entities, plan/region dimensions, month time dimension, simple MRR movement metrics, ratio/derived NRR and churn metrics.
  9. Operations: saved executive query, freshness gate, anomaly monitoring, artifact-based alerting.
Subscription metric composition
# Conceptual advanced metrics
metrics:
  - name: gross_mrr_churn_rate
    type: ratio
    numerator: churned_mrr
    denominator: beginning_mrr

  - name: net_revenue_retention
    type: derived
    expr: (beginning_mrr + expansion_mrr - contraction_mrr - churned_mrr) / nullif(beginning_mrr, 0)
    input_metrics:
      - beginning_mrr
      - expansion_mrr
      - contraction_mrr
      - churned_mrr
