Delta Lake Overview

Parquet data files plus a transactional metadata layer.

Data

Stored primarily in columnar Parquet files.

Transaction log

The _delta_log records every committed table change.

Capabilities

ACID, schema enforcement, UPDATE, DELETE, MERGE, time travel, and file-layout optimization.

Memory rule: Delta Lake = Parquet + transaction log + table-management features.
Delta table directory ├── part-00001-....parquet ├── part-00002-....parquet └── _delta_log/ ├── 00000000000000000000.json ├── 00000000000000000001.json └── checkpoint files

Plain Parquet vs Delta Update

How an update differs when deletion vectors are enabled.

Plain ParquetDelta Lake with a deletion vector
Updating one row usually requires rewriting the affected Parquet file through application logic.The original file can remain physically unchanged; the old row is marked as logically removed.
The updated value is stored in the rewritten data file.The updated row is written to a new file; Delta combines the base file, deletion vector, and new row during reads.
Advantage: Simple file format with little table metadata.Advantage: Faster row-level updates/deletes and ACID commit semantics.
Disadvantage: No built-in transaction log; partial or concurrent overwrite workflows can create inconsistent table state.Disadvantage: More metadata and read-path complexity; removed rows remain physically present until files are rewritten and later vacuumed.
Initial: 1.parquet contains customer_id 1–100 Update: customer_id = 1, quantity 5 → 10 Current Delta view = 1.parquet + deletion vector: skip old row for customer_id 1 + 2.parquet: customer_id 1, quantity 10
Important: Delta does not always use deletion vectors. When disabled or unsupported for an operation, Delta can rewrite affected files using copy-on-write behavior.
Enable deletion vectors
ALTER TABLE sales
SET TBLPROPERTIES ('delta.enableDeletionVectors' = 'true');

Deletion Vectors

Row-level deletion metadata that avoids immediately rewriting an entire Parquet file.

Memory rule: Deletion vector = keep the Parquet file, mark selected row positions as logically removed.

How it works

Step What Delta Lake does
1. Find affected rows Delta identifies the row positions affected by an UPDATE, DELETE, or MERGE.
2. Mark old rows A deletion vector records which rows in the existing Parquet file must be ignored.
3. Write replacement rows For an update, the new row values are written to a new Parquet file. For a delete, no replacement row is required.
4. Commit metadata The transaction log records the file's new logical state and references the deletion-vector information.
5. Read current snapshot Readers scan the base files, skip rows marked by deletion vectors, and include replacement rows from newer files.
Before UPDATE 2.parquet: customer_id 1 quantity 5 customer_id 2 quantity 7 customer_id 99 quantity 4 UPDATE customer_id = 1 SET quantity = 10 ↓ 2.parquet remains physically present + deletion vector: ignore old row for customer_id 1 + 3.parquet: customer_id 1, quantity 10 DELETE customer_id = 99 ↓ same deletion vector is extended: ignore row positions for customer_id 1 and 99

Transaction-log view

Operation Log actions Result
UPDATE Logical remove of the file's previous state, followed by add of the same file with a deletion vector, plus add of the file containing the updated row. The old row is skipped and the new row becomes visible atomically.
DELETE Logical remove of the previous file state and add of the file with an updated deletion vector. The deleted row becomes invisible without immediately rewriting the data file.

Enable deletion vectors

Enable on an existing Delta table
ALTER TABLE sales
SET TBLPROPERTIES (
  'delta.enableDeletionVectors' = 'true'
);
Check table properties
SHOW TBLPROPERTIES sales;

Advantages and drawbacks

Advantages Drawbacks
Faster sparse updates and deletes because affected files need not be rewritten immediately. Reads must apply deletion-vector metadata, which adds some processing complexity.
Reduces write amplification and can improve concurrency for row-level changes. Old rows still occupy physical storage until files are rewritten and obsolete files are vacuumed.
Works with Delta ACID transactions, snapshot isolation, history, and time travel. Requires compatible Delta readers and writers that support the deletion-vector table feature.

Deletion vectors, OPTIMIZE, and VACUUM

Deletion vector = logically hides rows OPTIMIZE = can rewrite files so deleted rows are physically removed from new active files VACUUM = later deletes obsolete physical files after the retention period
Do not confuse the steps: a deletion vector performs the immediate logical row removal; OPTIMIZE reorganizes or rewrites files; VACUUM permanently deletes eligible obsolete files.
Copy-on-Write Deletion-vector approach
Rewrite the affected Parquet file during the row change. Keep the base file and record deleted row positions separately.
Higher write cost, simpler current-state files. Lower immediate write cost, more work in the read path until files are rewritten.

ACID Transactions

Delta commits table changes atomically through the transaction log.

PropertyMeaning in Delta Lake
AtomicityA transaction is fully committed or not visible at all.
ConsistencySchema, protocol, constraints, and table invariants are validated during writes.
IsolationReaders see a consistent snapshot; concurrent writers are checked for conflicts.
DurabilityCommitted actions are persisted in _delta_log and referenced data files.
A transaction becomes visible only after its commit file is successfully added to the log.

PCC vs OCC for Isolation

Delta Lake primarily uses optimistic concurrency control.

PCC — Pessimistic Concurrency ControlOCC — Optimistic Concurrency Control
Assumes conflicts are likely.Assumes conflicts are uncommon.
Locks data before changes; another transaction may wait.Transactions work independently on snapshots without broad locks.
Prevents conflicts before they occur.Validates conflicts when a writer attempts to commit.
May introduce blocking and deadlocks.No blocking, but a conflicting writer can fail and must retry.
OCC: T1 reads V10 ── changes files ── commit check ── success → V11 T2 reads V10 ── changes files ── commit check ── conflict → retry
OCC: work independently, validate at commit.

Delta Transaction Log and Table Versions

JSON commits describe how Parquet files and deletion vectors form every table version.

Core transaction-log actions

ActionPurpose
addAdds a data file to the active table snapshot, optionally with a deletion vector.
removeLogically removes a file or its previous state from the active snapshot. Physical deletion happens later.
metaDataStores the schema, partition columns, table properties, and table identifier.
protocolDefines the minimum reader and writer features required by the table.
commitInfoCaptures operation details such as CTAS, INSERT, UPDATE, DELETE, MERGE, OPTIMIZE, and metrics.

How Delta Lake builds table versions

Operation Transaction log Physical / logical change
CTAS 0.json Creates 1.parquet with the initial records and records the first table version.
INSERT 1.json Adds 2.parquet containing newly inserted records.
UPDATE customer_id = 1 2.json Adds a deletion vector to 2.parquet to ignore the old row and creates 3.parquet with the updated row.
DELETE customer_id = 99 3.json Updates the deletion vector on 2.parquet so IDs 1 and 99 are ignored; no full file rewrite is required.
INSERT 4.json Creates 4.parquet for the next inserted records.
Current table snapshot = active Parquet files + deletion-vector rules + ordered actions from the Delta transaction log
Inspect table history
DESCRIBE HISTORY sales;
A remove action is normally logical. The old physical file can remain for time travel until it becomes eligible for VACUUM.

Handling Massive Delta Log Metadata

Delta reduces transaction-log replay and metadata overhead through compaction, checkpoints, and caching.

Mechanism What happens Benefit
JSON log compaction Multiple commit files such as 00000.json–00006.json can be represented by a compacted log file such as 00001.00006.compacted.json where supported. Reduces the number of small JSON files Delta must open and process.
Checkpointing At version 36, Delta writes a checkpoint such as 00036.checkpoint.parquet containing the complete active table state up to that version. To build version 38, Delta reads the checkpoint plus 00037.json and 00038.json, rather than replaying every commit from version 0.
Caching Frequently used table metadata, snapshots, and file lists can be retained in memory. Repeated queries avoid rebuilding the same table snapshot and reduce planning latency.
Read table at version 38 ↓ Load checkpoint at version 36 + Replay 00037.json + Replay 00038.json ↓ Construct version 38 snapshot

Commit JSON

Incremental changes for one or more versions.

Checkpoint

A materialized table-state snapshot used as a faster starting point.

Cache

Reuses recently computed metadata and snapshots for repeated access.

Latest snapshot = newest usable checkpoint + only the JSON commits written after that checkpoint.

Versioning & Time Travel

Every successful table-changing transaction creates a new version.

VersioningTime travel
Records successive committed table states: V0, V1, V2…Reads or restores a previous state by version or timestamp.
Useful for auditability and reproducibility.Useful for rollback, debugging, comparisons, and recovery.
SQL time travel and restore
DESCRIBE HISTORY sales;
SELECT * FROM sales VERSION AS OF 12;
SELECT * FROM sales TIMESTAMP AS OF '2026-07-10 10:00:00';
RESTORE TABLE sales TO VERSION AS OF 12;
Retention warning: After eligible old files are physically removed by VACUUM, versions that depend on them can no longer be read.

Schema-on-Read vs Schema-on-Write

Delta emphasizes schema-on-write through enforcement.

Schema-on-ReadSchema-on-Write
Store data first; interpret structure during reading.Validate data against a known schema before commit.
Flexible for raw and semi-structured landing zones.Produces consistent, trusted table structure.
Faster ingestion but more query-time complexity.Additional write validation but simpler downstream use.
Invalid or inconsistent records can remain in storage.Incompatible data is rejected unless evolution is enabled.

Common validations

Column namesData typesNullabilityRequired columnsExtra columnsNested structures
Schema enforcement = reject incompatible writes. Schema evolution = allow controlled compatible change.

Schema Evolution

Commands for adding columns, nested fields, widening types, and metadata-only changes.

TypeMeaningEnable / command
Add new columnsAdds compatible source columns; historical rows return NULL.SET spark.databricks.delta.schema.autoMerge.enabled = true; or per write.
Widen data typesAllows compatible widening such as INT → BIGINT.ALTER TABLE t SET TBLPROPERTIES ('delta.enableTypeWidening'='true');
Nested evolutionAdds compatible fields inside nested structures.Use automatic or per-write schema evolution where supported.
Column positionChanges display/order metadata using FIRST or AFTER.No session SET; use ALTER TABLE ... ALTER COLUMN.
Rename / dropMetadata-only rename/drop with column mapping.ALTER TABLE t SET TBLPROPERTIES ('delta.columnMapping.mode'='name');
Session-wide automatic evolution
SET spark.databricks.delta.schema.autoMerge.enabled = true;
Per-operation SQL evolution
INSERT WITH SCHEMA EVOLUTION INTO target_table
SELECT * FROM source_table;
Manual schema changes
ALTER TABLE target_table ADD COLUMNS (email STRING);
ALTER TABLE target_table ALTER COLUMN email AFTER customer_name;
ALTER TABLE target_table SET TBLPROPERTIES ('delta.enableTypeWidening' = 'true');
ALTER TABLE target_table ALTER COLUMN customer_id TYPE BIGINT;
ALTER TABLE target_table SET TBLPROPERTIES ('delta.columnMapping.mode' = 'name');
ALTER TABLE target_table RENAME COLUMN email TO email_address;
ALTER TABLE target_table DROP COLUMN email_address;
Best practice: Prefer per-write evolution when possible. Session-wide autoMerge can unintentionally accept schema changes from unrelated jobs.

Convert Parquet to Delta

In-place conversion creates a transaction log around existing Parquet files.

PySpark
from delta.tables import DeltaTable

DeltaTable.convertToDelta(
    spark,
    "parquet.`/mnt/data/customers`"
)
PySpark — partitioned Parquet
from delta.tables import DeltaTable

DeltaTable.convertToDelta(
    spark,
    "parquet.`/mnt/data/customers`",
    "event_date DATE"
)
SQL equivalent
SQL
CONVERT TO DELTA parquet.`/mnt/data/customers`;
CREATE TABLE customers USING DELTA LOCATION '/mnt/data/customers';
Operational rule: Stop writes during conversion. Afterwards, use Delta readers and writers.

Managed vs External Delta Tables

The distinction is who controls the data location and lifecycle.

Managed Delta TableExternal Delta Table
Created without a user-supplied LOCATION.Created with a user-supplied LOCATION.
The platform/catalog manages metadata and storage lifecycle.The catalog manages registration; storage remains independently controlled.
Dropping allows managed data cleanup according to platform lifecycle behavior.Dropping removes registration, while external files remain.
Best for centrally governed platform-managed data.Best for shared, portable, or independently retained storage.
Managed table
CREATE TABLE employees (
  employee_id INT,
  employee_name STRING,
  salary DOUBLE
)
USING DELTA;
External table

  employee_name STRING,
  salary DOUBLE

Copy-on-Write vs Merge-on-Read

A file-layout trade-off between write cost and read complexity.

Copy-on-Write (COW)Merge-on-Read-like behavior with deletion vectors
Updates/deletes rewrite affected files.Old rows can be marked by a deletion vector; changed rows are written separately.
Writes are more expensive.Writes are usually faster for sparse row changes.
Reads are straightforward.Reads combine base files with deletion-vector information and replacement rows.
Good for read-heavy workloads with fewer changes.Good for frequent UPDATE, DELETE, MERGE, and row-level concurrency.
Delta can use both patterns: file rewrite without deletion vectors; deferred row removal with deletion vectors.

Shallow Clone vs Deep Clone

Fast metadata reference versus fully independent copy.

Shallow CloneDeep Clone
Copies metadata and references source data files.Copies metadata and physical data files.
Fast and storage efficient.Slower and uses additional storage.
Depends on referenced source files.Fully independent after cloning.
Can break if referenced source files are removed.Unaffected by later source cleanup.
Best for test/dev and temporary environments.Best for backup, migration, and independent copies.
Clone commands
CREATE TABLE customer_test SHALLOW CLONE customer_prod;
CREATE TABLE customer_backup DEEP CLONE customer_prod;
Shallow = copy metadata, reference data. Deep = copy metadata + physical data.

CTAS vs Deep Clone

Derived query result versus near-exact independent table copy.

CTAS — CREATE TABLE AS SELECTDeep Clone
Creates a table from query output.Creates an independent copy at a selected source version.
Supports filtering, joining, projection, aggregation, transformation.Copies all source rows plus supported schema and table metadata.
Does not automatically preserve every property or constraint.Preserves more source structure and properties.
Best for derived datasets.Best for backup, migration, and exact-copy use cases.
CTAS
CREATE TABLE active_customers
USING DELTA AS
SELECT customer_id, name
FROM customers
WHERE status = 'ACTIVE';
Deep clone
CREATE TABLE customers_backup
DEEP CLONE customers;
CTAS = copy query result. Deep clone = copy table data + supported metadata.

Root Causes of the Small-File Problem

Many tiny files increase file-listing, metadata, task-scheduling, and scan overhead even though the data remains correct.

Root cause Example Why it creates small files
Frequent small batch writes Appending a few MB every minute. Each write creates one or more new files even when the incoming dataset is tiny.
Streaming micro-batches A trigger runs every 30 seconds. Every micro-batch can create separate output files.
Too many Spark output partitions df.repartition(10000) before writing 10 GB. Each output partition can create a file, producing roughly 1 MB files in this example.
High-cardinality table partitioning PARTITIONED BY (customer_id). Creates thousands or millions of directories with little data in each partition.
Low data volume per partition Hourly partitions receive only a few records. Even a reasonable partition column produces tiny files when each partition receives too little data.
Frequent UPDATE, DELETE, or MERGE Small CDC changes applied every few minutes. Changed rows or replacement data can be written into new small files that accumulate over time.
Multiple concurrent writers Several jobs append to the same table. Each job and task independently produces output files.
Incorrect repartitioning or coalescing Using an unnecessarily large partition count. The write has too little data per Spark partition to produce healthy file sizes.
Frequent writes + small input batches + too many output partitions + high-cardinality table partitioning + repeated row-level changes = many small Parquet files

Query impact

More file opens, more scan tasks, and slower query startup.

Metadata impact

Larger file lists and more transaction-log metadata.

Main fixes

Optimized writes, auto compaction, sensible partition counts, and periodic OPTIMIZE.

Compact accumulated small files
OPTIMIZE customer_table;

OPTIMIZE and Bin Packing

Compacts small files into fewer appropriately sized files.

BeforeAfter OPTIMIZE
Thousands of small files.Fewer, larger, balanced files.
High listing and task overhead.Lower metadata and scheduling overhead.
Logical data is unchanged.Logical data remains unchanged; physical layout changes.
Compaction commands
OPTIMIZE customer_table;

OPTIMIZE customer_table
WHERE order_date >= '2026-07-01';
10 MB + 20 MB + 15 MB + 25 MB + 30 MB ↓ bin packing fewer well-sized Parquet files
Not one giant file: OPTIMIZE creates multiple balanced files so Spark can keep parallel processing.

Optimized Writes vs Auto Compaction

Prevent small files during a write versus repair them after a write.

Optimized WritesAuto Compaction
Works during the write.Runs after a successful write.
Shuffles and groups output before files are created.Combines eligible small files that already exist.
Prevents many small files.Repairs remaining small files.
May increase write latency.Adds post-write processing.
Session settings
SET spark.databricks.delta.optimizeWrite.enabled = true;
SET spark.databricks.delta.autoCompact.enabled = auto;
Table properties
ALTER TABLE customer_table
SET TBLPROPERTIES (
  'delta.autoOptimize.optimizeWrite' = 'true',
  'delta.autoOptimize.autoCompact' = 'auto'
);
Optimized write = during file creation. Auto compaction = after file creation. OPTIMIZE = larger maintenance compaction.

VACUUM

Permanently removes old data files no longer referenced by the current table state.

CommandPurpose
VACUUM sales;Deletes eligible obsolete files using retention policy.
VACUUM sales DRY RUN;Lists eligible files without deleting them.
VACUUM sales RETAIN 168 HOURS;Retains seven days of obsolete files.
VACUUM sales FULL;Full eligible-file discovery where supported.
VACUUM sales LITE;Uses transaction-log information where supported.
UPDATE / DELETE / MERGE / OPTIMIZE ↓ Old files remain for time travel ↓ VACUUM after retention ↓ Old physical files permanently removed
OPTIMIZEVACUUM
Rewrites files to improve layout.Deletes obsolete files.
Improves query performance.Reduces storage use.
Creates new files; old files remain temporarily.Removes eligible old files permanently.
Safety: Short retention periods can break long-running readers/writers and remove time-travel capability. Use DRY RUN.

Z-Order

Maps multiple clustering dimensions into an approximate one-dimensional ordering.

StepWhat happens
Choose columnsUse frequently filtered columns such as product_id and order_date.
Map dimensionsRows are treated as multidimensional points; bit interleaving creates a 1D Z-value.
Preserve localityPoints close in multiple dimensions are often placed near each other.
Shuffle, sort, rewriteOPTIMIZE rewrites files so similar values are colocated.
Data skippingPer-file min/max statistics allow unrelated files to be skipped.
Z-Order command
OPTIMIZE sales
ZORDER BY (product_id, quantity);
(product_id, quantity) ↓ Z-order mapping 1-dimensional locality-preserving order ↓ shuffle + sort + file rewrite similar values stored in fewer files ↓ better data skipping
Clarification: Z-Order does not create table partitions. It reorganizes files inside existing table partitions.

Liquid Clustering

Flexible, incremental clustering without rigid partition directories.

FeatureExplanation
No fixed foldersData is organized by keys without hard-coded partition directories.
Flexible keysClustering columns can change as query patterns evolve.
Incremental reclusteringOPTIMIZE reclusters data that needs reorganizing.
Compaction + clusteringCreates fewer well-sized files and colocates similar key values.
Parallel processing remainsIt does not send all data to one core or create one file.
Not compatible with Z-OrderUse liquid clustering or Z-Order, not both.
Create clustered table
CREATE TABLE sales (
  product_id BIGINT,
  quantity INT,
  order_date DATE
)
USING DELTA
CLUSTER BY (product_id, order_date);

OPTIMIZE sales;
Change keys
ALTER TABLE sales
CLUSTER BY (customer_id, order_date);

OPTIMIZE sales;
Automatic keys where supported
ALTER TABLE sales
CLUSTER BY AUTO;

How to choose clustering keys

Choose columns that…Avoid columns that…
Frequently appear in selective WHERE filters.Are rarely filtered.
Have medium-to-high cardinality such as IDs and timestamps.Have only a few values such as booleans.
Match important repeated query patterns.Duplicate a highly correlated selected key.
Have data-skipping statistics.Add unnecessary dimensions; too many keys dilute effectiveness.
Liquid clustering = well-sized files + similar key values together + keys that can evolve.
