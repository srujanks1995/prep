30-second architecture answer

Snowflake separates durable cloud storage from independent virtual warehouses and a managed cloud-services layer. Storage is shared, compute is isolated by warehouse, and metadata, optimization, security and transactions are coordinated by cloud services.

Production answer framework

Observe → classify → verify → change → measure. Start with Query Profile, history and object metadata; identify the bottleneck; make one targeted change; compare elapsed time, bytes scanned, spill, queueing and credits.

Cloud servicesAuthentication, access control, metadata, parsing, optimization, transactionsMany metadata commands need no running warehouse; services also consume billable resources in some cases.
Why is Snowflake described as a hybrid shared-disk/shared-nothing architecture?
Storage is centrally shared like shared-disk systems, while each virtual warehouse is an independent compute cluster resembling shared-nothing execution. This gives common data access without compute contention across warehouses.
What happens when two warehouses query the same table?
They read the same durable storage but use separate compute and separate warehouse-local caches. One warehouse cannot consume the CPU or memory of the other.
Can Snowflake run on-premises?
No. Snowflake is a managed cloud service on supported public-cloud regions; infrastructure and software maintenance are managed by Snowflake.
Which layer performs query optimization?
The cloud-services layer parses and optimizes the query; the selected warehouse executes the plan, unless a reusable persisted result can be returned without recomputation.

Common interview trap: owning a table inside a managed access schema does not allow the table owner to independently grant access. Grant decisions are centralized with the schema owner or a role holding MANAGE GRANTS.
Why use separate schemas such as RAW, CORE and MART?
They communicate lifecycle and responsibility: RAW preserves source fidelity, CORE standardizes reusable business entities, and MART exposes consumption-ready dimensional models. Schema boundaries also simplify grants, ownership and deployment.
What is a managed access schema?
A schema in which object owners cannot make independent grant decisions. The schema owner or a role with MANAGE GRANTS controls current and future object privileges.
What are future grants?
Privilege templates applied to new objects of a specified type in a database or schema. They reduce manual grants but must be designed carefully around ownership and managed access behavior.

Scale up

Resize X-Small → Small → Medium. Best when individual queries need more compute or memory.

Scale out

Add clusters to a multi-cluster warehouse. Best when queueing is caused by concurrent queries.

Cold start

Resuming a warehouse clears its local data cache; do not benchmark a warm and cold run as equivalent.

Does doubling warehouse size always halve runtime?
No. Improvement depends on parallelism, pruning, join shape, remote spilling and compilation. A poorly filtered query can remain expensive, and small queries may not use extra capacity.
Standard versus Economy scaling policy?
Standard prioritizes minimizing queueing by starting clusters more readily. Economy conserves credits by favoring fuller utilization before adding clusters, potentially increasing latency.
Why separate BI and ETL warehouses?
Isolation avoids concurrency and cache disruption, permits different scaling and timeout settings, improves chargeback and lets each workload be tuned independently.
What causes queued overload time?
The warehouse has insufficient available execution slots for concurrent queries. Investigate concurrency, long-running blockers, warehouse size, multi-cluster settings and MAX_CONCURRENCY_LEVEL.

5. Micro-partitions, Clustering and Pruning

↑ Top
Core concept: Snowflake automatically writes table data into immutable, compressed, columnar micro-partitions and records metadata such as value ranges. Query pruning skips partitions whose metadata proves they cannot match the filter.
TermMeaningInterview use
Micro-partitionContiguous unit of table storage managed automaticallyNo user-created partitions or partition maintenance.
PruningSkipping unneeded micro-partitions and columnsPrimary reason selective filters run quickly.
ClusteringHow values are co-located across micro-partitionsGood clustering increases pruning for repeated selective patterns.
Clustering keyExpression used by automatic clusteringUse only for large tables with valuable, consistent filter/join patterns.
Clustering depthOverlap among partitions for key rangesHigher overlap can indicate poorer pruning, but interpret with workload context.
Clustering diagnostics
-- Inspect clustering quality
SELECT SYSTEM$CLUSTERING_INFORMATION(
  'analytics.mart.fct_events',
  '(event_date, customer_id)'
);

-- Enable a clustering key only after proving value
ALTER TABLE analytics.mart.fct_events
  CLUSTER BY (TO_DATE(event_ts), customer_id);
Why can wrapping a filtered column in a function hurt pruning?
The optimizer may be unable to directly use stored min/max metadata for the original column. Prefer sargable range predicates, such as timestamp boundaries, when possible.
Should every large table have a clustering key?
No. Automatic clustering adds serverless maintenance cost and storage churn. Use it only when pruning benefits recurring high-value queries enough to justify the cost.
How do updates affect micro-partitions?
Micro-partitions are immutable. Updates and deletes create new partition versions and metadata rather than modifying blocks in place; Time Travel can retain prior versions.
What is a better filter: DATE(order_ts)=... or a timestamp range?
A half-open timestamp range is usually clearer for pruning: order_ts >= start and order_ts < next_day. It also avoids time-of-day boundary mistakes.

Why is the second execution faster?
Possible reasons include persisted result reuse, warm warehouse data cache or warm metadata/compilation cache. Disable persisted results and compare Query Profile before attributing the gain.
Does resizing a warehouse preserve its data cache?
Do not rely on it. Resize/suspend operations can replace compute resources and invalidate the local cache. Treat warehouse cache as opportunistic, not durable.
When is result cache invalidated?
When relevant data or query conditions change, or reuse requirements are not met. It should not be treated as a substitute for physical tuning or modeled aggregates.

Do not say: “Transient tables have no Time Travel.” They can have Time Travel, but they do not have Fail-safe. Temporary tables are session scoped and can create Information Schema overhead if created excessively.
Permanent versus transient table?
Permanent tables add Fail-safe protection after Time Travel, while transient tables omit Fail-safe and are appropriate for reproducible data where lower recovery coverage is acceptable.
External table versus Iceberg table?
An external table exposes files through Snowflake metadata, commonly as read-oriented external data. Iceberg tables use the open Iceberg table format with snapshots and catalog semantics, enabling broader interoperability and richer table behavior.
Hybrid table versus standard table?
Hybrid tables target low-latency row operations and enforce primary keys, with indexes and different storage behavior. Standard tables remain the default for large-scale analytical scans and micro-partition-based workloads.
Regular versus materialized view?
A regular view stores SQL and executes against current base data. A materialized view persists and automatically maintains eligible results, trading maintenance cost for faster repeated access.

Grain-first modeling
-- Fact grain: one row per order line
CREATE TABLE mart.fct_order_line (
  order_line_key NUMBER,
  order_id       NUMBER,
  line_number    NUMBER,
  order_date_key NUMBER,
  customer_key   NUMBER,
  product_key    NUMBER,
  quantity       NUMBER(12,2),
  gross_amount   NUMBER(18,2),
  discount_amount NUMBER(18,2),
  net_amount     NUMBER(18,2),
  load_ts        TIMESTAMP_LTZ
);

-- Protect the declared grain
SELECT order_id, line_number, COUNT(*)
FROM mart.fct_order_line
GROUP BY 1,2
HAVING COUNT(*) > 1;
What is grain and why state it first?
Grain defines exactly what one row represents. Keys, measures, deduplication and joins are correct only when the grain is explicit.
When would you choose a star schema in Snowflake?
For analytics and BI where understandable relationships, fewer joins and stable metric behavior matter more than minimizing dimension duplication.
Should you fully normalize because Snowflake handles joins well?
Not automatically. Compute elasticity does not remove semantic complexity, fan-out risk or BI usability concerns. Model for workload, governance and consumers.
How do you model multiple fact tables?
Keep each fact at its natural grain, share conformed dimensions and avoid directly joining facts unless they are aggregated to a compatible grain or connected through a deliberate bridge.

WHERE versus QUALIFY?
WHERE filters rows before grouping/window evaluation. QUALIFY filters after window functions, avoiding a subquery for patterns such as latest row per key.
Why use TRY_TO_NUMBER in ingestion?
It quarantines malformed values as NULL rather than failing the whole statement. Production logic should still count and route rejected values instead of silently accepting them.
How do you make latest-row logic deterministic?
Use a complete ORDER BY with a business timestamp plus a stable tie-breaker such as ingestion timestamp, source sequence or file/row metadata.

Should all JSON stay in one VARIANT column?

What is the difference between SQL NULL and JSON null?
They are distinct concepts in VARIANT handling. Use functions such as IS_NULL_VALUE when the distinction matters.
Why can repeated FLATTEN be expensive?
It expands data at query time and can multiply rows. For recurring access, parse once into a modeled table or consider an eligible materialized representation.

12. Bulk Loading with COPY INTO

↑ Top
Auditable COPY pattern
COPY INTO raw.orders
FROM (
  SELECT
    t.$1::NUMBER AS order_id,
    TRY_TO_TIMESTAMP_NTZ(t.$2) AS order_ts,
    t.$3::NUMBER(18,2) AS amount,
    METADATA$FILENAME AS source_file,
    METADATA$FILE_ROW_NUMBER AS source_row,
    CURRENT_TIMESTAMP() AS load_ts

FROM @raw.orders_stage
VALIDATION_MODE = 'RETURN_ERRORS';

File sizePrefer a manageable set of reasonably sized compressed filesToo many tiny files add overhead; huge files reduce parallelism and recovery granularity.
IdempotencyUse load history plus business dedupe keysSnowflake tracks loaded files, but renamed/changed files and downstream reruns need design.

COPY INTO versus INSERT?
COPY is optimized for staged file loading, supports file formats, load history, validation and file metadata. INSERT is suited to row/set operations from SQL sources, not bulk file ingestion.
Why did COPY skip a file?
Snowflake load history may consider it already loaded. Verify file path/name, last modified behavior, target table history and whether FORCE was intentionally used.
Should you use FORCE=TRUE to rerun loads?
Only deliberately. It can create duplicates. Prefer immutable file names, batch manifests and idempotent downstream MERGE logic.
How do you handle partial failures?
Capture COPY results, route rejects to a quarantine workflow, reconcile source counts/control totals, correct the cause and replay only the affected batch.

Auto-ingest integrates with cloud event notifications. The notification reaches Snowflake ingestion infrastructure, and the named pipe executes its stored COPY definition.

Check cloud event delivery and supported event type, queue/integration configuration, pipe status, stage path overlap, privileges, file format, load history and COPY errors.

14. Streams and Change Data Capture

↑ Top
A stream stores an offset, not a duplicate copy of all changed rows. Querying exposes change records between the offset and current table version. Consuming it in committed DML advances the offset.
CDC consumption pattern
CREATE OR REPLACE STREAM core.orders_changes
  ON TABLE core.orders;

-- METADATA$ACTION: INSERT or DELETE
-- METADATA$ISUPDATE pairs delete/insert records for updates
MERGE INTO mart.orders_current t
USING (
  SELECT *
  FROM core.orders_changes
  QUALIFY ROW_NUMBER() OVER (
    PARTITION BY order_id
    ORDER BY updated_at DESC
  ) = 1
) s
ON t.order_id = s.order_id
WHEN MATCHED AND s.METADATA$ACTION = 'DELETE' THEN DELETE
WHEN MATCHED THEN UPDATE SET
  status = s.status,
  updated_at = s.updated_at
WHEN NOT MATCHED AND s.METADATA$ACTION = 'INSERT' THEN
  INSERT (order_id, status, updated_at)
  VALUES (s.order_id, s.status, s.updated_at);
Stream typeCapturesTypical source
StandardInserts, deletes and update pairsSnowflake tables/views/dynamic tables where supported.
Append-onlyInserted rows only, optimized when deletes/updates are irrelevantInsert-only event/landing patterns.
Insert-onlyExternal-table change semanticsExternal table ingestion scenarios.
Does SELECT from a stream consume it?
A plain SELECT does not advance the offset. The offset advances when stream data is used by a committed DML transaction.
Can two consumers share one stream?
They would share one offset and interfere. Create one stream per independent consumer.
What is stream staleness?
If the source history required by the stream offset ages beyond retention, the stream can become stale. Monitor staleness and consume within the retention window.
How are updates represented?
A standard stream exposes an update as a DELETE record for the old row and an INSERT record for the new row, marked with METADATA$ISUPDATE.

16. Dynamic Tables and Refresh Semantics

↑ Top
Declarative refresh
CREATE OR REPLACE DYNAMIC TABLE mart.daily_sales
  TARGET_LAG = '15 minutes'
  WAREHOUSE = transform_wh
  REFRESH_MODE = INCREMENTAL
  INITIALIZE = ON_CREATE
AS
SELECT
  order_date,
  region,
  SUM(net_amount) AS revenue,
  COUNT(DISTINCT order_id) AS orders
FROM core.order_line
GROUP BY 1,2;

ALTER DYNAMIC TABLE core.enriched_orders
  SET TARGET_LAG = DOWNSTREAM;

Dynamic tables versus streams and tasks?
Dynamic tables are declarative: define the desired SELECT and freshness. Streams/tasks are imperative and better when you need procedural control, side effects, custom transaction boundaries or unsupported transformations.
Target lag versus schedule?
Target lag specifies how fresh output should be. Snowflake chooses refresh timing; it does not promise execution exactly every N minutes.
Why is a dynamic table behind target lag?
Check refresh history, warehouse capacity, upstream lag, query complexity, refresh mode, data-change volume, policy compatibility and whether refresh duration approaches the target.
Why set intermediate target lag to DOWNSTREAM?
It lets downstream freshness requirements drive the pipeline and avoids independently refreshing intermediate tables that no consumer currently needs.
What consistency does a dynamic table pipeline provide?
Refreshes operate on a consistent point-in-time view of inputs; linked dynamic tables are coordinated as a pipeline.

How would you refresh a customer dimension every 15 minutes?
For file arrivals, Snowpipe into raw plus a dynamic table or stream/triggered task into the dimension. Choose dynamic tables for declarative SQL; choose streams/tasks for precise SCD logic and custom auditing.
How do you guarantee a dashboard never sees a half-loaded batch?
Load into staging, validate counts and controls, then publish atomically using a transaction, table swap, batch-status predicate or view pointer. Consumers should filter to committed batches only.
How do you replay a failed day?
Retain immutable raw input and batch metadata, isolate the failed batch, correct code/configuration, rerun into a temporary target, reconcile and publish idempotently without reprocessing unrelated successful data.

19. MERGE, Deduplication and SCD Patterns

↑ Top
SCD Type 1
-- SCD Type 1: latest source record wins
MERGE INTO core.customer_current t
USING (
  SELECT *
  FROM stage.customer_delta
  QUALIFY ROW_NUMBER() OVER (
    PARTITION BY customer_id
    ORDER BY source_updated_at DESC, ingestion_ts DESC
  ) = 1
) s
ON t.customer_id = s.customer_id
WHEN MATCHED AND HASH(t.name, t.segment) <> HASH(s.name, s.segment)
  THEN UPDATE SET
    name = s.name,
    segment = s.segment,
    source_updated_at = s.source_updated_at,
    updated_at = CURRENT_TIMESTAMP()
WHEN NOT MATCHED THEN INSERT (
  customer_id, name, segment, source_updated_at, created_at, updated_at
) VALUES (
  s.customer_id, s.name, s.segment, s.source_updated_at,
  CURRENT_TIMESTAMP(), CURRENT_TIMESTAMP()
);
SCD Type 2 two-step concept
-- SCD Type 2 concept
-- 1) close the current changed row
UPDATE core.dim_customer d
SET valid_to = s.effective_ts,
    is_current = FALSE
FROM stage.customer_changes s
WHERE d.customer_id = s.customer_id
  AND d.is_current
  AND d.row_hash <> s.row_hash;

-- 2) insert new current versions
INSERT INTO core.dim_customer (..., valid_from, valid_to, is_current, row_hash)
SELECT ..., effective_ts, '9999-12-31', TRUE, row_hash
FROM stage.customer_changes s
WHERE NOT EXISTS (
  SELECT 1 FROM core.dim_customer d
  WHERE d.customer_id = s.customer_id
    AND d.is_current
    AND d.row_hash = s.row_hash
);
Why deduplicate before MERGE?
Multiple source rows matching one target row can produce nondeterministic behavior or errors. Resolve precedence explicitly using business timestamps and tie-breakers.
SCD Type 1 versus Type 2?
Type 1 overwrites current attributes and keeps no history. Type 2 closes the current version and inserts a new version with effective dates and a current flag.
How do you process late-arriving SCD2 records?
Use event-effective time rather than load time, locate the version interval containing the event, split/resequence intervals and revalidate non-overlap. This requires more than a simple current-row update.
How do you make MERGE idempotent?
Use stable business/change identifiers, deterministic source dedupe and conditions that only update when data changed. Reprocessing the same batch should make no further changes.

36. Detailed Scenario-Based Interview Questions

↑ Top
Scenario: A dashboard became slow after data doubled. What do you do?
  1. Capture slow query IDs and compare profiles before/after.
  2. Check partitions scanned, pruning ratio, join row explosion, spill and queueing.
  3. Confirm filters still align with date/key access patterns and statistics.
  4. Fix SQL/grain first; then test clustering, aggregate/dynamic table, search optimization or warehouse sizing.
  5. Compare latency and credits under representative concurrency.
Scenario: Queries are fast individually but slow at 9 AM.
This indicates concurrency rather than single-query capacity. Review queued overload time and arrival pattern; isolate BI from ETL, enable/tune multi-cluster scaling, stagger refreshes and reduce redundant dashboard queries.
Scenario: A Snowpipe load stopped after a cloud change.
Check event notification type/destination, integration trust, queue subscription, stage URL/path, pipe status, ownership/privileges and COPY errors. Verify a test object event reaches the queue before changing Snowflake SQL.
Scenario: The target has duplicate rows after a retry.
Separate ingestion idempotency from transformation idempotency. Identify repeated files/batches, dedupe source by stable change ID, make MERGE deterministic, add a unique-grain test and store successful batch checkpoints.
Scenario: A stream is empty although the table changed.
Verify the stream offset may already have advanced in committed DML, changes occurred before stream creation/offset, the right source object is used, append-only semantics, retention/staleness and whether a different consumer shared the same stream.
Scenario: A dynamic table misses its 15-minute target lag.
Inspect refresh history and upstream lag, refresh duration, warehouse contention, change volume, refresh mode and unsupported/expensive operators. Set intermediates to DOWNSTREAM, optimize incremental logic or increase capacity only after diagnosis.
Scenario: A task is resumed but never executes.
Check whether it is a child whose root is suspended, the schedule/WHEN condition, stream data, owner-role privileges, warehouse/serverless configuration, predecessor state and task history.
Scenario: A MERGE raises multiple-match errors.
The source has multiple rows for a target key or target key is not unique. Profile both sides, dedupe source with deterministic ROW_NUMBER, enforce grain tests and clean target duplicates before rerun.
Scenario: A selective customer lookup still scans most partitions.
Customer IDs may be scattered across micro-partitions and clustering may not match. Verify predicate form and Query Profile; evaluate Search Optimization for equality lookup or a model optimized for the access pattern.
Scenario: Warehouse cost rose but query volume is flat.
Break cost into runtime, size, cluster count and idle time. Look for longer queries, spills, reduced pruning, auto-suspend changes, scheduled overlap, credit-heavy retries and serverless feature spend that may not appear in warehouse usage.
Scenario: A resource monitor did not stop Snowpipe cost.
Resource monitors control user-managed virtual warehouses, not all serverless services. Use budgets and feature-specific usage views for Snowpipe and other serverless spend.
Scenario: Analysts can see a table but SELECT fails.
They may have object SELECT but lack USAGE on the database/schema, be using the wrong active role, or a policy may deny rows. Inspect grants through the role hierarchy and test with the actual role.
Scenario: Row-level security leaks data through a view.
Review whether the policy is attached to the correct base/view object and its mapping logic, role hierarchy checks, secure view usage and owner/caller context. Test negative cases with consumer roles and inspect policy references.
Scenario: A cloned production database exposes PII in dev.
Zero-copy clone preserves data and attached metadata/policies but dev role mappings or policy dependencies may differ. Use a governed clone process that masks/subsets data and validates policy references before granting access.
Scenario: A table was accidentally dropped.
Within Time Travel retention, use UNDROP. If a new object with the same name exists, rename it first. Validate dependent objects and privileges after recovery; Fail-safe is not the first operational mechanism.
Scenario: Cross-region failover succeeded but the app cannot connect.
DR includes client routing. Validate connection URLs, Client Redirect or DNS, authentication integrations, network policies, secrets and role/object replication—not only database promotion.
Scenario: External-table queries are much slower than native tables.
Inspect external bytes/files scanned and partition pruning, file format/size and metadata refresh. Materialize hot data into native tables or Iceberg where appropriate, and keep only cold/exploratory access external.
Scenario: JSON queries are expensive.
Promote frequently used paths to typed columns, filter before FLATTEN, avoid repeated expansion, cluster/search-optimize only after evidence and materialize commonly flattened structures.
Scenario: A new source column broke downstream jobs.
Land schema drift safely in raw, alert on contract change, preserve previous interface, add/convert the column in staging, run compatibility tests and migrate downstream consumers through an expand/contract release.
Scenario: Updates to a large table create high storage cost.
Review update pattern, Time Travel retention, clustering rewrite, transient/permanent choice and whether immutable append + latest-state model is better. Large UPDATE/DELETE operations create new micro-partition versions.
Scenario: A query uses DISTINCT and is still wrong.
DISTINCT may hide duplicate rows but cannot repair incorrect measures after fan-out. State the fact grain, validate dimension key uniqueness, aggregate before joining and remove the incorrect many-to-many path.
Scenario: Same SQL is fast in prod but slow in QA.
Compare warehouse size/state/cache, data volume and clustering, object definitions, statistics/partitions, active role/policies, parameter settings and concurrency. Do not attribute it to environment name alone.
Scenario: BI refresh and ELT collide every hour.
Use separate warehouses and coordinate schedules. Isolate service-level objectives, set query tags/timeouts, stagger expensive transformations and scale each workload independently.
Scenario: Dynamic table or streams/tasks—which do you pick?
Choose dynamic tables for a declarative SELECT and freshness objective. Choose streams/tasks when you need procedural branching, custom MERGE, multiple targets, API/procedure side effects or exact transaction control.
Scenario: Build a near-real-time sales pipeline.
Snowpipe Streaming or event connector → immutable raw events → typed/deduped staging → dynamic tables or stream-triggered MERGE into core → mart aggregates → quality/audit tables. Define watermark, late-event handling, replay, SLA and cost monitoring.
Scenario: Design tenant-isolated analytics.
Include tenant_id in every relevant grain, enforce row access policies or secure views, separate high-risk domains if required, use tenant-aware role/database-role mapping, test caching/sharing behavior, audit access and prevent cross-tenant joins.
Scenario: Interviewer asks “How would you optimize Snowflake?”
Answer in order: workload/SLA → Query History/Profile → SQL and grain → pruning/data layout → warehouse capacity/concurrency → cache/materialization → cost controls → measure before/after. Avoid listing features without diagnosis.

What is a virtual warehouse?
An independent cluster of compute resources used to execute queries, DML, loading and transformations.
Does a warehouse store table data?
No. Durable table data is in centralized cloud storage; the warehouse may hold a temporary local cache.
What is auto-suspend?
Automatic warehouse suspension after an idle period to stop compute credit use.
Scale up versus scale out?
Scale up increases resources per cluster; scale out adds clusters for concurrency.
What is a micro-partition?
An automatically managed immutable columnar storage unit with metadata used for pruning.
Can users define partitions?
Not like traditional partitioned databases; Snowflake automatically creates micro-partitions.
What is partition pruning?
Skipping micro-partitions whose metadata proves they cannot satisfy the query predicate.
What is clustering depth?
A measure of overlap across micro-partitions for clustering-key ranges.
What is result cache?
A persisted prior query result that can be reused when validity conditions are met.
What is warehouse cache?
Local data cached on a running warehouse’s compute resources.
What is a transient table?
A persistent table without Fail-safe, intended for reproducible data.
What is a temporary table?
A session-scoped table dropped at session end.
What is a dynamic table?
A declaratively defined table whose result Snowflake refreshes to meet target freshness.
What is target lag?
The desired maximum freshness lag relative to upstream data, not a fixed cron schedule.
What is a stream?
A CDC offset object exposing changes to a source object since the offset.
Does SELECT consume a stream?
No; committed DML consumption advances the offset.
What is a triggered task?
A task that runs when a stream has new data instead of on a fixed schedule.
What is Snowpipe?
Serverless continuous file ingestion using a pipe’s COPY statement.
What is Snowpipe Streaming?
Low-latency row/event ingestion without first writing batch files to a stage.
What is a stage?
A Snowflake object/location used to store or reference files for loading and unloading.
What is a file format?
A reusable definition of how Snowflake parses or writes file bytes.
What does COPY history do?
Tracks file load outcomes and supports auditing/troubleshooting of staged-file loads.
What is VARIANT?
A type for semi-structured values including JSON objects, arrays and scalars.
What does FLATTEN do?
Expands arrays/objects into rows as a table function.
What does QUALIFY do?
Filters rows after window functions are evaluated.
What is zero-copy cloning?
Instant cloning by initially sharing immutable micro-partitions with the source.
What is Time Travel?
User-accessible historical query and recovery within configured retention.
What is Fail-safe?
A Snowflake-managed last-resort recovery period for permanent data after Time Travel.
What is a secure view?
A view designed to protect definition details and reduce information leakage.
What is data sharing?
Provider-controlled access to selected objects without exporting/copying data to the consumer.
What is a reader account?
A provider-managed Snowflake account for consumers without their own account.
What is a database role?
A role scoped to one database that packages object privileges and is granted to account roles.
What is a managed access schema?
A schema where grant decisions are centralized with the schema owner/MANAGE GRANTS.
What is a masking policy?
A policy that dynamically transforms a column’s returned value by context.
What is a row access policy?
A policy that determines whether each row is visible.
Which policy is evaluated first?
Row access is evaluated before masking when both apply.
What is Search Optimization Service?
A maintained access-path service for supported highly selective lookup/search predicates.
What is automatic clustering?
A serverless service that maintains clustering for tables with clustering keys.
What is a materialized view?
A persisted, automatically maintained eligible query result used for performance.
What is query spill?
Intermediate data exceeding memory and writing to local or remote storage.
What is queued overload time?
Time waiting because the warehouse cannot run all concurrent queries.
What is QUERY_TAG?
Session/query metadata used for attribution, monitoring and chargeback.
What is a resource monitor?
A credit monitor/control for user-managed virtual warehouses.
What is a Snowflake budget?
A broader cost-monitoring and notification construct for supported resources, including serverless usage.
What is an external table?
Snowflake table metadata over files that remain in external storage.
What is an Iceberg table?
A table using Apache Iceberg format and metadata, typically in external object storage.
What is a hybrid table?
A Snowflake table optimized for low-latency row access with enforced keys and indexes.
What is Snowpark?
Developer APIs that build relational computations executed in Snowflake.
UDF versus procedure?
A UDF participates in expressions; a procedure orchestrates multi-statement logic via CALL.
What isolation level is used?
READ COMMITTED for standard transaction behavior.
Are PK/FK constraints enforced on standard tables?
Generally no, except NOT NULL; they are metadata/optimizer information. Hybrid tables enforce keys.
What is RPO?
Maximum acceptable data loss measured in time.
What is RTO?
Maximum acceptable service restoration time.
Replication group versus failover group?
Replication copies supported objects; failover groups add promotion/failback capability.
Why use separate warehouses?
Workload isolation, independent scaling, cache behavior, SLA and cost attribution.
Why avoid ACCOUNTADMIN for pipelines?
It violates least privilege and increases blast radius; use dedicated scoped roles.
How do you find expensive queries?
Use Query History/ACCOUNT_USAGE and inspect Query Profile, bytes, partitions, spill, queue and credits.
How do you find failed tasks?
Use TASK_HISTORY and inspect owner role, schedule/condition, warehouse and error details.
What is schema evolution?
Controlled adaptation of a table to compatible incoming schema changes during supported loads.
What is INFER_SCHEMA?
A function that derives column definitions from supported staged files.
What is an SCD Type 2 dimension?
A dimension retaining history with effective intervals and current-version markers.
How do you prevent MERGE duplicates?
Deterministically dedupe source and test target business-key uniqueness.
What is fan-out?
Row multiplication from a many-to-many or nonunique join that corrupts aggregates.
What is a conformed dimension?
A shared dimension definition used consistently across multiple fact tables/marts.
What is a fact table grain?
The exact business event or snapshot represented by one fact row.
What is Cortex Search?
Managed hybrid semantic and keyword retrieval with reranking for search/RAG applications.
Why evaluate AI outputs?
Model outputs are nondeterministic; measure retrieval relevance, groundedness, safety, latency and cost.
What is the fastest performance fix?
There is no universal one—diagnose with evidence before choosing SQL, storage or compute changes.

38. End-to-End Design Cases

↑ Top

Case A — Batch enterprise warehouse

Requirement: 2 TB/day files, hourly SLA, Tableau dashboards, reproducible loads.

Design: cloud storage → external stage/storage integration → scheduled COPY into immutable RAW → staged quality/dedupe → streams/tasks or dbt into CORE → star-schema MART → dedicated multi-cluster BI warehouse. Capture batch/file metadata, query tags, quality controls and resource budgets.

Trade-off: Snowpipe is possible, but scheduled bulk COPY may be simpler and more cost-efficient for large predictable hourly drops.

Case B — Near-real-time customer events

Requirement: seconds-to-minutes latency, Kafka, late events, customer-360 metrics.

Design: Kafka connector/Snowpipe Streaming → raw append-only events with event_id and source sequence → dedupe/watermark → dynamic table pipeline or stream-triggered MERGE → current state + immutable event fact → incremental aggregates. Track event time and processing time separately.

Trade-off: dynamic tables reduce orchestration; streams/tasks offer more control for late-event corrections and side effects.

Case C — Secure multi-tenant analytics product

Requirement: customers access governed reports; no cross-tenant leakage.

Design: tenant key in all facts/dimensions → row access policy or secure per-product views → database roles per package/tier → secure share/listing → policy and negative-security tests → access history audit. Separate compute for internal and customer workloads.

Trade-off: a single shared model is efficient but demands rigorous policy correctness; physical separation increases cost/operations but can reduce blast radius.

Case D — Cross-region disaster recovery

Requirement: RPO 15 minutes, RTO 60 minutes.

Design: failover groups containing databases and required account objects → frequent monitored refresh → client redirect/connection strategy → runbook for promotion, validation and failback → quarterly drills. Measure actual refresh lag and application cutover, not only Snowflake object status.
