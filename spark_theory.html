Why Spark and the Core Mental Model

Spark is a distributed processing engine designed to execute multi-step data pipelines as a DAG instead of forcing every pipeline into repeated map and reduce jobs.

ConceptMeaning / UseInterview perspective
MapReduce limitationIntermediate results are repeatedly materialized to durable storage between jobs; complex pipelines become chains of separate MapReduce jobs.Explain disk and orchestration overhead, not simply “MapReduce is slow.”
Spark improvementPipelines narrow operations together, keeps reusable data in memory when useful, and materializes mainly at shuffle or output boundaries.Spark is not “always in memory”; shuffle spill and output still use disk.
DAG modelTransformations form a dependency graph. An action asks Spark to optimize and execute the required path.Connect DAG → job → stages → tasks.

Common interview questions

  • Why can Spark outperform classic MapReduce?
  • Does Spark avoid disk completely?
  • What is the difference between a DAG and a stage?

Jobs, Stages, Tasks, and Partitions

This hierarchy is the most important Spark interview mental model. Actions create jobs, shuffle boundaries divide jobs into stages, and partitions determine task count.

Key Concepts: 1 Action → 1 Job 1 Job → 2 Stages: one for non-shuffle (map), one for shuffle (reduceByKey) Each stage → Tasks (1 task per partition) Default partition size = 128 MB → num_partitions = file_size / 128 MB Default shuffle partitions = 200 (shuffle data is written to disk first) Flow: Partitions → Tasks → sent to Executors
ConceptMeaning / UseInterview perspective
JobExecution created by an action. A single user action can sometimes produce supporting jobs as well.Look in the Jobs and SQL tabs.
StageA set of tasks that can run without another shuffle boundary.Wide transformations commonly introduce boundaries.
TaskSmallest scheduled unit; processes one partition for one stage.Too few tasks underuse cores; too many tiny tasks add overhead.
PartitionLogical chunk of distributed data and the basis of parallel work.Partition count is not executor count.

Common interview questions

  • If a stage has 500 partitions, how many tasks are created?
  • Why can one action appear as multiple jobs?
  • Where are stage boundaries created?

Spark Architecture

The driver owns the application plan and coordination. Executors perform distributed work. The cluster manager allocates resources but does not execute dataframe logic.

Spark Architecture Driver Program └── SparkContext / SparkSession └── Cluster Manager (YARN / Mesos / Kubernetes / Standalone) ├── Executor 1 [cores: parallelism] [memory: storage + execution] │ ├── Task 1 │ ├── Task 2 │ └── Task N └── Executor 2 [cores: parallelism] [memory: storage + execution] ├── Task 1 └── Task N Key Points: Driver: plans the DAG, schedules tasks, tracks job progress Executor: runs tasks, stores cached data Parallelism = number of cores across all executors Memory = --executor-memory split into: spark.memory.fraction (execution + storage) Reserved memory (300 MB) Each core runs one task at a time
ConceptMeaning / UseInterview perspective
DriverCreates plans, coordinates stages, tracks metadata and task results.Driver OOM often comes from collect, large metadata, or too many files/tasks.
ExecutorLong-lived application process that runs tasks and can cache data.Executor loss triggers recomputation from lineage when possible.
Cluster managerAllocates CPU and memory resources.YARN, Kubernetes, and Standalone are deployment choices.
Worker nodeMachine hosting one or more executor processes.A worker is not the same thing as an executor.

Common interview questions

  • Driver vs executor responsibilities?
  • What happens when an executor fails?
  • Why is total cores the upper bound for simultaneous tasks?

RDD vs DataFrame vs Dataset

The APIs differ mainly in structure, optimization visibility, type safety, and language support.

Memory layout: RDD: stores JVM objects → GC pressure, slow serialization DataFrame/Dataset: Tungsten stores data off-heap in binary format → no GC, fast Rule of thumb: Prefer DataFrame in Python (PySpark). Use RDD only when you need low-level partition control or work with unstructured data.

Partitioning vs Bucketing

Partitioning creates directory groups by column values. Bucketing hashes rows into a fixed number of buckets. They solve different physical-layout problems.

ConceptMeaning / UseInterview perspective
Physical layoutPartitioning: directories by value.Bucketing: fixed hash buckets.
Best columnsPartitioning: low/moderate-cardinality columns such as date or region.Bucketing: high-cardinality join/group keys such as customer_id.
PruningPartition pruning can skip directories.Bucket pruning can skip compatible buckets for supported filters.
Join impactPartitioning may help only when join filters align with partition keys.Compatible bucketing can reduce or avoid join shuffle.
Main riskToo many directories and tiny files.Wrong bucket count, skewed keys, or metadata incompatibility.
Combined designPartition by date and bucket by customer_id.Useful when queries filter by date and join by customer.

Repartition vs Coalesce

Both change partition count, but they have different data-movement behavior and use cases.

8b. repartition vs coalesce repartition(n) coalesce(n) Shuffle ✅ Full shuffle ❌ No shuffle (merge) Direction Increase or decrease Only decrease Use when Fixing data skew, slow stages Fixing small file problem
ConceptMeaning / UseInterview perspective
repartitionRedistributes records and can increase or decrease partitions.Use for balanced downstream parallelism or distribution by keys.
coalesceUsually merges existing partitions without a full shuffle.Use to reduce small output files when data is already reasonably balanced.

Use repartition(n) to control number of output part files

Shuffle: What Happens and Why It Is Expensive

Shuffle redistributes records so related keys reach the same downstream partition. It costs network I/O, serialization, sorting, memory, disk spill, and synchronization.

ConceptMeaning / UseInterview perspective
Shuffle writeMap-side tasks partition, serialize, buffer, sort, and write blocks.Spill occurs when buffers exceed memory.
Shuffle fetchReduce-side tasks fetch blocks from many executors.Network and remote-block failures can dominate.
Shuffle readFetched records are merged, aggregated, sorted, or joined.Skew creates a few very large tasks.
Stage barrierNext stage waits for required upstream shuffle output.One slow task can delay the entire stage.

Spark Join Strategies — Interview View

Spark selects a physical join based on predicate type, join type, size estimates, hints, existing distribution, and adaptive runtime statistics.

ConceptMeaning / UseInterview perspective
Broadcast hash joinOne equi-join side is small enough to copy to every executor.Broadcast → build hash table → probe large side. Avoids large-side shuffle.
Sort-merge joinBoth equi-join sides are large and sortable.Shuffle both sides → sort → merge. Stable for large data but shuffle-heavy.
Shuffled hash joinBoth sides shuffle; one side of each resulting partition can fit in memory.Build local hash table per partition. Faster than sorting when safe, but memory-sensitive.
Broadcast nested-loop joinNon-equi/range/cross condition and one side can be broadcast.Broadcast small side, then compare pairs. Potentially very expensive.
Cartesian productNo usable equality condition and no broadcast strategy.Avoid unless intentionally producing all combinations.
Physical-plan keywords to recognize: BroadcastHashJoin, SortMergeJoin, ShuffledHashJoin, BroadcastNestedLoopJoin, BroadcastExchange, Exchange hashpartitioning, BuildLeft, BuildRight, AdaptiveSparkPlan.

Catalyst Optimizer and Physical Planning

Catalyst turns unresolved expressions into an analyzed logical plan, rewrites it, creates physical alternatives, and selects an executable strategy.

Phases of Execution in Spark User Code (transformations + action) ↓ 1. DAG Creation — Driver builds logical execution plan (lazy) ↓ 2. Logical Plan — Unresolved → Analyzed → Optimized (Catalyst Optimizer) ↓ 3. Physical Plan — Catalyst picks best physical strategy (sort-merge join vs broadcast join etc.) ↓ 4. Code Generation — Tungsten generates optimized JVM bytecode (whole-stage codegen) ↓ 5. Job Submission — Action triggers job; DAGScheduler splits DAG into Stages ↓ 6. Stage → TaskSet — Each stage broken into tasks (1 task per partition) ↓ 7. Task Scheduling — TaskScheduler assigns tasks to Executors via Cluster Manager ↓ 8. Task Execution — Executors run tasks; shuffle write at stage boundary ↓ 9. Shuffle Read — Next stage reads shuffle files from disk ↓ 10. Result returned — Final result sent to Driver (collect) or written to sink (write) Key components: Catalyst Optimizer: logical plan → optimized logical plan (predicate pushdown, column pruning, constant folding) Tungsten: memory management + code generation for CPU efficiency DAGScheduler: splits DAG at shuffle boundaries into stages TaskScheduler: assigns tasks to free executor slots Stage boundary = every wide transformation (shuffle)
ConceptMeaning / UseInterview perspective
AnalyzerResolves tables, columns, data types, and functions using catalogs.Unresolved plan is syntactic; analyzed plan is semantically resolved.
Rule optimizerApplies rewrites such as constant folding and predicate simplification.Rules do not require runtime data.
Cost-based planningUses statistics to compare physical alternatives.Poor statistics can lead to poor choices.
Physical planContains operators such as scans, exchanges, sorts, aggregates, and joins.Read physical plans from child scans upward.

Common interview questions

  • Logical vs physical plan?
  • What does Catalyst optimize?
  • Why can estimated statistics matter?

Spark Memory Model

Executor memory includes JVM heap plus additional container memory. Within heap, Spark balances execution, storage, user objects, and reserved space.

ConceptMeaning / UseInterview perspective
Execution memoryShuffle, sort, join, and aggregation working memory.Can evict storage blocks when needed.
Storage memoryCached partitions and some broadcast blocks.Borrowing is dynamic within unified memory.
User memoryApplication objects, UDF objects, metadata, and non-Spark structures.Not governed by the unified pool.
Reserved memoryInternal safety region.Not available for normal workload use.
Memory overheadNon-heap JVM, native, Python workers, and system overhead.Container can be killed even when JVM heap is below its limit.
Off-heapOptional memory outside JVM heap.Reduces GC pressure but requires correct sizing and monitoring.

Executor Sizing and Memory Allocation

Executor sizing balances concurrency, memory per task, garbage collection, failure blast radius, and cluster utilization.

Q11. Memory Allocation Setup (example: 64 GB node, 16 cores, 2 executors) Node RAM = 64 GB OS overhead = 10% → 6.4 GB Usable RAM = 64 - 6.4 = 57.6 GB Executors = 2 (= num_cores / cores_per_executor = 16 / 8) Memory/executor = 57.6 / 2 = ~28 GB (--executor-memory) JVM overhead = 10% of executor memory → ~2.8 GB (spark.executor.memoryOverhead) Spark memory = 28 - 2.8 = ~25 GB Unified pool = Spark memory × spark.memory.fraction (0.6 default) = 25 × 0.6 = 15 GB Reserved buffer = Spark memory × 0.4 = 10 GB (user code, internal metadata) Storage pool = Unified pool × spark.memory.storageFraction (0.6) = 15 × 0.6 = 9 GB ← cache/persist Execution pool = Unified pool × 0.4 = 15 × 0.4 = 6 GB ← shuffle/sort/join Cores/executor = 16 / 2 = 8 → 8 tasks run in parallel per executor Formulas usable_ram = node_ram × 0.9 memory_per_exec = usable_ram / num_executors spark_memory = memory_per_exec × (1 - overhead_fraction) # overhead default 0.1 unified_pool = spark_memory × spark.memory.fraction # default 0.6 storage_pool = unified_pool × spark.memory.storageFraction # default 0.6 execution_pool = unified_pool × (1 - storageFraction) # default 0.4 cores_per_exec = total_cores / num_executors When to tune Symptom Fix Shuffle spill / OOM on join Increase execution pool → lower storageFraction Cache evicted too often Increase storage pool → raise storageFraction Many small files, low CPU Add more executors (smaller --executor-memory, more instances) CPU >90% and OOM together Add more nodes to the cluster Long GC pauses Reduce executor memory or switch to off-heap (spark.memory.offHeap.enabled) Dynamic allocation spark.dynamicAllocation.enabled = true spark.dynamicAllocation.minExecutors = 1 spark.dynamicAllocation.maxExecutors = 10 spark.dynamicAllocation.initialExecutors = 2 Spark scales executor count up/down automatically based on pending tasks.
ConceptMeaning / UseInterview perspective
Too many cores/executorMore simultaneous tasks share one heap and one GC process.Can amplify memory pressure and long GC pauses.
Too few cores/executorMore executors and overhead; potentially underuses node capacity.Balance concurrency and failure isolation.
Large executorsMore cache room but larger GC domain and bigger loss on failure.Avoid “one giant executor per node” by default.
Small executorsBetter isolation and parallelism but more overhead.Useful for elastic and heterogeneous workloads.

Common interview questions

  • Why not use all node memory?

Spark and Kafka — Interview View

Kafka partitions determine source parallelism and per-partition order. Spark checkpoints track consumed offsets and state progress.

ConceptMeaning / UseInterview perspective
Source parallelismLimited by Kafka partitions unless Spark splits work further internally.More executors cannot exceed useful source parallelism.
OrderingGuaranteed only within a Kafka partition.No global ordering across partitions.
OffsetsRepresent source progress.Checkpoint continuity is essential after restart.
BackpressureProcessing must keep up with arrival rate.Tune ingestion rate, partitions, cluster capacity, and state cost.
