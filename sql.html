01. SQL Foundations & Query Order

Build correct queries by understanding logical execution order, NULL behavior, CASE, aliases, and safe calculations.

7 patterns
coreMedium

Logical SQL execution order

Remember the order used to reason about aliases, aggregation, and filtering.

Pattern: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
-- Written order
SELECT department, AVG(salary) AS avg_salary
FROM employees
WHERE status = 'Active'
GROUP BY department
HAVING AVG(salary) > 80000
ORDER BY avg_salary DESC
LIMIT 5;
WHERE filters rows before grouping; HAVING filters groups after aggregation.
coreEasy

SELECT, aliases, DISTINCT

Choose columns, rename output, and remove duplicate result rows.

Pattern: SELECT [DISTINCT] expression AS alias FROM table
SELECT DISTINCT
       department AS dept
FROM employees
ORDER BY dept;
Most databases allow SELECT aliases in ORDER BY, but not in WHERE.
filterEasy

WHERE with multiple conditions

Combine conditions carefully and use parentheses when AND and OR appear together.

Pattern: WHERE (condition_a OR condition_b) AND condition_c
SELECT *
FROM orders
WHERE (status = 'Shipped' OR status = 'Delivered')
  AND order_date >= DATE '2026-01-01';
AND is evaluated before OR. Parentheses make intent explicit.
nullEasy

NULL: IS NULL, COALESCE, NULLIF

Handle missing values and division by zero safely.

Pattern: IS NULL | COALESCE(value, fallback) | NULLIF(a, b)
SELECT employee_id,
       COALESCE(bonus, 0) AS bonus,
       revenue / NULLIF(quantity, 0) AS unit_revenue
FROM employee_sales
WHERE manager_id IS NULL;
Never compare NULL using = or <>. Use IS NULL / IS NOT NULL.
caseEasy

CASE WHEN classification

Create categories or conditional values inside a query.

Pattern: CASE WHEN condition THEN value ... ELSE value END
SELECT employee_id,
       salary,
       CASE
         WHEN salary >= 100000 THEN 'High'
         WHEN salary >= 60000  THEN 'Medium'
         ELSE 'Low'
       END AS salary_band
FROM employees;
Conditions are checked top-to-bottom; put the most specific rule first.
nullMedium

Safe percentage calculation

Protect calculations from missing values and zero denominators.

Pattern: 100.0 * numerator / NULLIF(denominator, 0)
SELECT category,
       ROUND(
         100.0 * (current_sales - previous_sales)
         / NULLIF(previous_sales, 0),
         2
       ) AS growth_pct
FROM category_growth;
Use 100.0 rather than 100 when your database may perform integer division.
dmlEasy

Swap values in one column

Use CASE in UPDATE to exchange two values safely.

Pattern: UPDATE table SET col = CASE WHEN ... END
UPDATE employees
SET status = CASE
  WHEN status = 'Active'   THEN 'Inactive'
  WHEN status = 'Inactive' THEN 'Active'
  ELSE status
END;
Always preserve unexpected values with ELSE status.

02. Aggregation & GROUP BY

Master grouped calculations, conditional aggregation, HAVING, ratios, pivoting, and string aggregation.

9 patterns
aggregateEasy

GROUP BY with multiple metrics

Calculate count, average, minimum, maximum, and total per group.

Pattern: GROUP BY grouping_columns
SELECT department,
       COUNT(*) AS employee_count,
       AVG(salary) AS avg_salary,
       MIN(salary) AS min_salary,
       MAX(salary) AS max_salary,
       SUM(salary) AS payroll
FROM employees
GROUP BY department;
Every selected non-aggregated column must appear in GROUP BY.
aggregateEasy

WHERE vs HAVING

Use WHERE for row-level filtering and HAVING for aggregate-level filtering.

Pattern: WHERE rows → GROUP BY → HAVING groups
SELECT department, AVG(salary) AS avg_salary
FROM employees
WHERE status = 'Active'
GROUP BY department
HAVING AVG(salary) > 80000;
Push row filters into WHERE for correctness and usually better performance.
duplicatesEasy

Find duplicate values

Group by the business key and keep groups appearing more than once.

Pattern: GROUP BY key HAVING COUNT(*) > 1
SELECT email, COUNT(*) AS duplicate_count
FROM employees
GROUP BY email
HAVING COUNT(*) > 1;
For complete duplicate rows, group by every column that defines equality.
caseMedium

Conditional aggregation

Count or sum selected rows without splitting the query into many subqueries.

Pattern: SUM(CASE WHEN condition THEN 1 ELSE 0 END)
SELECT department,
       COUNT(*) AS total,
       SUM(CASE WHEN status = 'Active' THEN 1 ELSE 0 END) AS active_count,
       SUM(CASE WHEN status = 'Inactive' THEN 1 ELSE 0 END) AS inactive_count
FROM employees
GROUP BY department;
PostgreSQL also supports COUNT(*) FILTER (WHERE condition).
caseMedium

Customers who bought A and C, but not B

Translate set membership requirements into HAVING conditions.

Pattern: GROUP BY entity HAVING conditional counts
SELECT customer_id
FROM orders
GROUP BY customer_id
HAVING SUM(CASE WHEN product = 'A' THEN 1 ELSE 0 END) > 0
   AND SUM(CASE WHEN product = 'C' THEN 1 ELSE 0 END) > 0
   AND SUM(CASE WHEN product = 'B' THEN 1 ELSE 0 END) = 0;
This pattern generalizes to required, optional, and forbidden events.
aggregateMedium

Percentage of total

Divide a group total by the grand total using a window over aggregated results.

Pattern: SUM(metric) / SUM(SUM(metric)) OVER ()
SELECT department,
       SUM(salary) AS payroll,
       ROUND(
         100.0 * SUM(salary) / SUM(SUM(salary)) OVER (),
         2
       ) AS payroll_pct
FROM employees
GROUP BY department;
The inner SUM aggregates each department; the outer window SUM gets the grand total.
pivotMedium

Pivot rows into columns

Use conditional aggregation for a portable pivot.

Pattern: SUM(CASE WHEN category = value THEN metric ELSE 0 END)
SELECT department,
       SUM(CASE WHEN month_name = 'Jan' THEN revenue ELSE 0 END) AS jan_revenue,
       SUM(CASE WHEN month_name = 'Feb' THEN revenue ELSE 0 END) AS feb_revenue,
       SUM(CASE WHEN month_name = 'Mar' THEN revenue ELSE 0 END) AS mar_revenue
FROM department_revenue
GROUP BY department;
Static pivots require known output columns; dynamic pivots need database-specific SQL.
aggregateMedium

String aggregation

Combine grouped text values into an ordered comma-separated list.

Pattern: STRING_AGG(value, delimiter ORDER BY value)
SELECT sell_date,
       COUNT(DISTINCT product) AS num_sold,
       STRING_AGG(DISTINCT product, ',' ORDER BY product) AS products
FROM activities
GROUP BY sell_date
ORDER BY sell_date;
MySQL uses GROUP_CONCAT; SQL Server uses STRING_AGG without DISTINCT in some versions.
aggregateHard

Median by group

Use a percentile function when supported.

Pattern: PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY value)
SELECT department,
       PERCENTILE_CONT(0.5)
         WITHIN GROUP (ORDER BY salary) AS median_salary
FROM employees
GROUP BY department;
Percentile syntax varies by database; a ranking-based fallback may be required.

03. JOINs, Anti-JOINs & Self JOINs

Reason about matching, missing rows, duplicate multiplication, NULLs, manager relationships, and interval joins.

9 patterns
joinEasy

INNER vs LEFT JOIN

INNER keeps matches; LEFT keeps every left row plus matches from the right.

Pattern: A JOIN B ON matching_condition
SELECT e.employee_id,
       e.employee_name,
       d.department_name
FROM employees e
LEFT JOIN departments d
  ON d.department_id = e.department_id;
Columns from the right table become NULL when a LEFT JOIN finds no match.
anti-joinEasy

Find missing foreign-key matches

Return employees whose department row does not exist.

Pattern: LEFT JOIN ... WHERE right.key IS NULL
SELECT e.*
FROM employees e
LEFT JOIN departments d
  ON d.department_id = e.department_id
WHERE d.department_id IS NULL;
Filter on a non-nullable right-side key, not an arbitrary nullable column.
anti-joinMedium

NOT EXISTS anti-join

Find rows with no related match using a NULL-safe correlated subquery.

Pattern: WHERE NOT EXISTS (SELECT 1 FROM child WHERE ...)
SELECT c.customer_id, c.customer_name
FROM customers c
WHERE NOT EXISTS (
  SELECT 1
  FROM orders o
  WHERE o.customer_id = c.customer_id
);
NOT EXISTS is generally safer than NOT IN when the subquery can return NULL.
self-joinMedium

Employees earning more than their manager

Join a hierarchy table to itself using employee and manager aliases.

Pattern: employee.manager_id = manager.employee_id
SELECT e.employee_name,
       e.salary,
       m.employee_name AS manager_name,
       m.salary AS manager_salary
FROM employees e
JOIN employees m
  ON m.employee_id = e.manager_id
WHERE e.salary > m.salary;
Aliases make the two roles in the same table clear.
joinMedium

JOIN duplicate multiplication

Understand why duplicate keys create more result rows than expected.

Pattern: rows for key k = count_A(k) × count_B(k)
-- A has key 1 twice; B has key 1 three times.
-- INNER JOIN rows for key 1 = 2 × 3 = 6.
SELECT a.id, b.id
FROM a
JOIN b ON a.id = b.id;
A many-to-many join is valid only when that multiplication matches the business requirement.
joinMedium

NULL behavior in JOINs

Standard equality does not match NULL to NULL.

Pattern: NULL = NULL evaluates to UNKNOWN
SELECT *
FROM a
FULL OUTER JOIN b
  ON a.id = b.id;

-- To intentionally match NULLs in PostgreSQL:
-- ON a.id IS NOT DISTINCT FROM b.id
Do not assume NULL rows from both sides will match in an equality join.
join mathHard

joinHard

Non-equi join: effective-dated lookup

Match a fact row to the dimension version active on that date.

Pattern: fact_date >= valid_from AND fact_date < valid_to
SELECT f.order_id,
       f.order_date,
       d.customer_tier
FROM orders f
JOIN customer_history d
  ON d.customer_id = f.customer_id
 AND f.order_date >= d.valid_from
 AND f.order_date < COALESCE(d.valid_to, DATE '9999-12-31');
Use half-open intervals [start, end) to avoid boundary overlap.
joinHard

Find overlapping intervals

Two intervals overlap when each starts before the other ends.

Pattern: a.start < b.end AND b.start < a.end
SELECT a.booking_id, b.booking_id
FROM bookings a
JOIN bookings b
  ON a.room_id = b.room_id
 AND a.booking_id < b.booking_id
 AND a.start_time < b.end_time
 AND b.start_time < a.end_time;
The ID inequality prevents pairing a row with itself and avoids duplicate pairs.

04. Subqueries, CTEs & Set Operations

Break complex logic into stages, compare sets, and traverse hierarchical data.

8 patterns
cteEasy

CTE for readable stages

Name intermediate results so each stage has one responsibility.

Pattern: WITH stage AS (...) SELECT ... FROM stage
WITH department_sales AS (
  SELECT department_id, SUM(amount) AS sales
  FROM orders
  GROUP BY department_id
)
SELECT *
FROM department_sales
WHERE sales > 100000;
A CTE improves readability; it does not automatically guarantee faster execution.
subqueryMedium

Correlated subquery

Evaluate a subquery using values from the current outer row.

Pattern: WHERE value > (SELECT ... WHERE inner.key = outer.key)
SELECT e.*
FROM employees e
WHERE e.salary > (
  SELECT AVG(e2.salary)
  FROM employees e2
  WHERE e2.department = e.department
);
A window function can often express the same logic with one scan.
subqueryMedium

EXISTS vs IN

Use EXISTS for relationship checks and IN for small explicit sets.

Pattern: EXISTS stops after a qualifying match
SELECT c.*
FROM customers c
WHERE EXISTS (
  SELECT 1
  FROM orders o
  WHERE o.customer_id = c.customer_id
    AND o.order_date >= CURRENT_DATE - INTERVAL '30 days'
);
NOT IN can return no rows when its subquery contains NULL; prefer NOT EXISTS.
setEasy

UNION vs UNION ALL

Combine compatible result sets with or without duplicate removal.

Pattern: UNION removes duplicates; UNION ALL preserves them
SELECT customer_id, 'web' AS source FROM web_orders
UNION ALL
SELECT customer_id, 'store' AS source FROM store_orders;
UNION adds a deduplication step and is usually more expensive.
setMedium

INTERSECT and EXCEPT

Compare sets directly when your database supports these operators.

Pattern: A INTERSECT B | A EXCEPT B
-- Customers present in both years
SELECT customer_id FROM orders_2025
INTERSECT
SELECT customer_id FROM orders_2026;

-- Customers lost in 2026
SELECT customer_id FROM orders_2025
EXCEPT
SELECT customer_id FROM orders_2026;
MySQL may require EXISTS / NOT EXISTS equivalents.
recursiveHard

Recursive employee hierarchy

Traverse all employees who report directly or indirectly to the CEO.

Pattern: anchor query UNION ALL recursive query
WITH RECURSIVE reports AS (
  SELECT employee_id, manager_id, 1 AS level
  FROM employees
  WHERE manager_id = 1

  UNION ALL

  SELECT e.employee_id, e.manager_id, r.level + 1
  FROM employees e
  JOIN reports r
    ON e.manager_id = r.employee_id
)
SELECT *
FROM reports
ORDER BY level, employee_id;
Use cycle detection or a depth limit when malformed hierarchies are possible.
recursiveHard

Find all ancestors

Walk upward from an employee through the management chain.

Pattern: start at employee → repeatedly join manager
WITH RECURSIVE chain AS (
  SELECT employee_id, manager_id, employee_name, 0 AS level
  FROM employees
  WHERE employee_id = 42

  UNION ALL

  SELECT m.employee_id, m.manager_id, m.employee_name, c.level + 1
  FROM employees m
  JOIN chain c
    ON m.employee_id = c.manager_id
)
SELECT * FROM chain;
The direction of the recursive join determines whether you walk upward or downward.
duplicatesMedium

Delete duplicates with CTE + ROW_NUMBER

Rank duplicates and delete every row except the selected keeper.

Pattern: PARTITION BY business_key ORDER BY keeper_rule
WITH ranked AS (
  SELECT employee_id,
         ROW_NUMBER() OVER (
           PARTITION BY email
           ORDER BY employee_id
         ) AS rn
  FROM employees
)
DELETE FROM employees
WHERE employee_id IN (
  SELECT employee_id FROM ranked WHERE rn > 1
);
Preview the ranked rows before deleting. Delete syntax differs across databases.

05. Window Functions

Rank rows, compare periods, calculate running and rolling metrics, and choose correct ROWS or RANGE frames without collapsing detail.

14 patterns
rankingEasy

ROW_NUMBER vs RANK vs DENSE_RANK

Choose unique numbering, competition ranking, or gapless ranking.

Pattern: ... OVER (PARTITION BY ... ORDER BY ...)
SELECT employee_name,
       salary,
       ROW_NUMBER() OVER (ORDER BY salary DESC) AS row_num,
       RANK()       OVER (ORDER BY salary DESC) AS rank_num,
       DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank_num
FROM employees;
ROW_NUMBER is always unique; RANK skips after ties; DENSE_RANK does not.
rankingMedium

Top 1 per department

Rank inside each department and keep the first row.

Pattern: ROW_NUMBER() OVER (PARTITION BY group ORDER BY metric DESC)
WITH ranked AS (
  SELECT e.*,
         ROW_NUMBER() OVER (
           PARTITION BY department
           ORDER BY salary DESC, employee_id
         ) AS rn
  FROM employees e
)
SELECT *
FROM ranked
WHERE rn = 1;
Add a deterministic tie-breaker after salary.
rankingMedium

Top N per group including ties

Use DENSE_RANK when all rows tied at the Nth value should remain.

Pattern: DENSE_RANK() ... WHERE rank <= N
WITH ranked AS (
  SELECT e.*,
         DENSE_RANK() OVER (
           PARTITION BY department
           ORDER BY salary DESC
         ) AS salary_rank
  FROM employees e
)
SELECT *
FROM ranked
WHERE salary_rank <= 3;
ROW_NUMBER returns exactly N rows per group; DENSE_RANK may return more.
windowEasy

LAG and LEAD

Read a previous or next row without a self join.

Pattern: LAG(value) | LEAD(value)
SELECT product_id,
       sale_month,
       sales,
       LAG(sales)  OVER (PARTITION BY product_id ORDER BY sale_month) AS prev_sales,
       LEAD(sales) OVER (PARTITION BY product_id ORDER BY sale_month) AS next_sales
FROM monthly_sales;
Ordering must uniquely define row sequence for deterministic results.
windowMedium

Month-over-month growth

Aggregate monthly sales, use LAG, then calculate protected growth.

Pattern: aggregate → LAG → percentage
WITH monthly AS (
  SELECT product_id,
         DATE_TRUNC('month', order_date) AS month_start,
         SUM(sales) AS sales
  FROM orders
  GROUP BY product_id, DATE_TRUNC('month', order_date)
), compared AS (
  SELECT monthly.*,
         LAG(sales) OVER (
           PARTITION BY product_id ORDER BY month_start
         ) AS previous_sales
  FROM monthly
)
SELECT *,
       ROUND(100.0 * (sales - previous_sales)
             / NULLIF(previous_sales, 0), 2) AS mom_pct
FROM compared;
Aggregate to the comparison grain before using LAG.
windowEasy

Running total

Accumulate all values from the beginning through the current row.

Pattern: ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
SELECT order_date,
       sales,
       SUM(sales) OVER (
         ORDER BY order_date
         ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS running_sales
FROM daily_sales;
Specify ROWS explicitly when duplicate ORDER BY values are possible.
window frameMedium

Window frame syntax and boundaries

Understand what the starting and ending boundaries mean before choosing a frame.

Pattern: ROWS BETWEEN frame_start AND frame_end
-- General form
aggregate_function(value) OVER (
  PARTITION BY group_column
  ORDER BY sort_column
  ROWS BETWEEN frame_start AND frame_end
)

-- Valid boundary keywords
-- UNBOUNDED PRECEDING : first row in the partition
-- n PRECEDING         : n physical rows before
-- CURRENT ROW         : the current physical row
-- n FOLLOWING         : n physical rows after
-- UNBOUNDED FOLLOWING : last row in the partition
The frame start must not come after the frame end. A deterministic ORDER BY is important when sort values repeat.
window frameMedium

Common ROWS frames and when to use them

Choose a frame based on whether the calculation is cumulative, backward-looking, centered, forward-looking, or partition-wide.

Pattern: Running | rolling | centered | forward | full-partition
SELECT transaction_id,
       amount,

       -- 1. Running total: first row through current row
       SUM(amount) OVER (
         ORDER BY transaction_id
         ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS running_total,

       -- 2. Rolling 3 rows: current row plus 2 previous rows
       AVG(amount) OVER (
         ORDER BY transaction_id
         ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
       ) AS rolling_3_row_avg,

       -- 3. Centered window: previous, current, and next row
       AVG(amount) OVER (
         ORDER BY transaction_id
         ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING
       ) AS centered_3_row_avg,

       -- 4. Forward-looking: current row plus next 2 rows
       SUM(amount) OVER (
         ORDER BY transaction_id
         ROWS BETWEEN CURRENT ROW AND 2 FOLLOWING
       ) AS next_3_row_total,

       -- 5. Entire partition: same total on every row
       SUM(amount) OVER (
         ROWS BETWEEN UNBOUNDED PRECEDING
                  AND UNBOUNDED FOLLOWING
       ) AS overall_total
FROM transactions;
Use ROWS when the requirement is expressed as a number of physical records, such as the previous 7 transactions—not the previous 7 calendar days.
window frameHard

ROWS versus RANGE with duplicate values

ROWS counts physical records; RANGE groups peer rows that have the same ORDER BY value.

Pattern: ROWS = records | RANGE = ORDER BY value or value interval
SELECT sale_id,
       sale_date,
       amount,

       -- Advances one physical row at a time.
       SUM(amount) OVER (
         ORDER BY sale_date, sale_id
         ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS rows_running_total,

       -- All rows having the same sale_date are peers.
       -- They receive the same cumulative value.
       SUM(amount) OVER (
         ORDER BY sale_date
         RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS range_running_total
FROM sales;

-- PostgreSQL time-based example:
-- RANGE BETWEEN INTERVAL '6 days' PRECEDING AND CURRENT ROW
Use ROWS for transaction-by-transaction calculations. Use RANGE for value-based peers or true time intervals where the SQL engine supports interval frames.
windowMedium

Rolling 3-row average

Calculate a moving metric over the current row and two prior rows.

Pattern: ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
SELECT order_date,
       sales,
       AVG(sales) OVER (
         ORDER BY order_date
         ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
       ) AS rolling_3_row_avg
FROM daily_sales;
A row-based frame is not the same as a 3-day time range when dates are missing.
windowHard

Rolling 30-day sum

Use a range frame where supported to model a time interval.

Pattern: RANGE BETWEEN INTERVAL '29 days' PRECEDING AND CURRENT ROW
SELECT order_date,
       sales,
       SUM(sales) OVER (
         ORDER BY order_date
         RANGE BETWEEN INTERVAL '29 days' PRECEDING AND CURRENT ROW
       ) AS rolling_30_day_sales
FROM daily_sales;
Window RANGE interval support differs across SQL engines.
windowMedium

FIRST_VALUE and LAST_VALUE

Get boundary values inside a partition, with an explicit frame for LAST_VALUE.

Pattern: FIRST_VALUE(value) | LAST_VALUE(value)
SELECT customer_id,
       order_date,
       amount,
       FIRST_VALUE(amount) OVER w AS first_amount,
       LAST_VALUE(amount)  OVER w AS last_amount
FROM orders
WINDOW w AS (
  PARTITION BY customer_id
  ORDER BY order_date
  ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
);
Default LAST_VALUE often ends at the current row, so specify the full frame.
windowMedium

NTILE for quartiles

Divide ordered rows into approximately equal buckets.

Pattern: NTILE(number_of_buckets)
SELECT employee_id,
       salary,
       NTILE(4) OVER (ORDER BY salary DESC) AS salary_quartile
FROM employees;
Buckets may differ by one row when the row count is not divisible evenly.

06. Date, Time & Retention Patterns

Normalize dates, compare periods, detect first and repeat activity, and calculate cohorts and retention.

8 patterns
dateEasy

Filter a date range safely

Use a half-open interval so timestamps on the final day are included correctly.

Pattern: timestamp >= start AND timestamp < next_period_start
SELECT *
FROM orders
WHERE order_ts >= TIMESTAMP '2026-01-01 00:00:00'
  AND order_ts <  TIMESTAMP '2026-02-01 00:00:00';
Avoid applying functions to an indexed timestamp column in the WHERE clause.
dateEasy

Aggregate by month

Convert timestamps to a consistent monthly grain before aggregation.

Pattern: DATE_TRUNC('month', timestamp)
SELECT DATE_TRUNC('month', order_date) AS month_start,
       SUM(amount) AS revenue
FROM orders
GROUP BY DATE_TRUNC('month', order_date)
ORDER BY month_start;
MySQL commonly uses DATE_FORMAT; SQL Server often uses DATEFROMPARTS/EOMONTH.
dateEasy

First order per customer

Use MIN when only the first date is needed.

Pattern: MIN(event_date) GROUP BY entity
SELECT customer_id,
       MIN(order_date) AS first_order_date
FROM orders
GROUP BY customer_id;
Use ROW_NUMBER when you need the full first-order row, not only the date.
retentionMedium

Logged in one day after first login

Compare later activity with each player’s first activity date.

Pattern: first_date + interval = event_date
WITH first_login AS (
  SELECT player_id, MIN(event_date) AS first_date
  FROM activity
  GROUP BY player_id
), retained AS (
  SELECT DISTINCT a.player_id
  FROM activity a
  JOIN first_login f
    ON f.player_id = a.player_id
   AND a.event_date = f.first_date + INTERVAL '1 day'
)
SELECT ROUND(
         1.0 * COUNT(*) /
         NULLIF((SELECT COUNT(*) FROM first_login), 0),
         2
       ) AS day_1_retention
FROM retained;
Deduplicate retained users before calculating the fraction.
dateMedium

Days between consecutive events

Use LAG to compare each event date with the previous one.

Pattern: event_date - LAG(event_date)
SELECT customer_id,
       order_date,
       order_date - LAG(order_date) OVER (
         PARTITION BY customer_id ORDER BY order_date
       ) AS days_since_previous_order
FROM orders;
Date subtraction syntax and output type vary by database.
cohortHard

Cohort month and activity month

Assign each user to a first-activity cohort and measure later monthly activity.

Pattern: cohort_month = month(MIN(event_date))
WITH user_events AS (
  SELECT user_id,
         DATE_TRUNC('month', event_date) AS activity_month,
         DATE_TRUNC(
           'month',
           MIN(event_date) OVER (PARTITION BY user_id)
         ) AS cohort_month
  FROM events
)
SELECT cohort_month,
       activity_month,
       COUNT(DISTINCT user_id) AS active_users
FROM user_events
GROUP BY cohort_month, activity_month
ORDER BY cohort_month, activity_month;
Add month offsets to create a classic retention matrix.
dateMedium

Year-over-year comparison

Aggregate yearly metrics and compare with the prior year inside each category.

Pattern: LAG(metric) OVER (PARTITION BY category ORDER BY year)
WITH yearly AS (
  SELECT category,
         EXTRACT(YEAR FROM order_date) AS order_year,
         SUM(sales) AS sales
  FROM orders
  GROUP BY category, EXTRACT(YEAR FROM order_date)
)
SELECT yearly.*,
       LAG(sales) OVER (
         PARTITION BY category ORDER BY order_year
       ) AS previous_year_sales
FROM yearly;
Missing calendar years require a date spine if comparisons must be truly consecutive.
dateHard

Generate a date spine

Create missing dates so zero-activity periods remain visible.

Pattern: generate_series(start, end, interval)
WITH calendar AS (
  SELECT d::date AS calendar_date
  FROM generate_series(
    DATE '2026-01-01',
    DATE '2026-01-31',
    INTERVAL '1 day'
  ) AS g(d)
)
SELECT c.calendar_date,
       COALESCE(SUM(o.amount), 0) AS revenue
FROM calendar c
LEFT JOIN orders o
  ON o.order_date::date = c.calendar_date
GROUP BY c.calendar_date
ORDER BY c.calendar_date;
PostgreSQL example; other engines use calendar tables or recursive CTEs.

07. String & Data-Cleaning SQL

Normalize messy text, parse fields, validate formats, and identify anomalies directly in SQL.

7 patterns
stringEasy

Trim and normalize case

Remove surrounding whitespace and standardize matching case.

Pattern: LOWER(TRIM(column))
SELECT customer_id,
       INITCAP(TRIM(customer_name)) AS clean_name,
       LOWER(TRIM(email)) AS clean_email
FROM customers;
Function names vary: INITCAP is not available in every engine.
stringEasy

Replace and concatenate

Build display values and safely combine nullable strings.

Pattern: CONCAT | CONCAT_WS | REPLACE
SELECT customer_id,
       CONCAT_WS(' ', first_name, middle_name, last_name) AS full_name,
       REPLACE(phone, '-', '') AS digits_only
FROM customers;
CONCAT_WS usually skips NULL values; || may propagate NULL in some engines.
stringEasy

Extract domain from email

Split a delimited string and select the required part.

Pattern: SPLIT_PART(value, delimiter, position)
SELECT email,
       SPLIT_PART(LOWER(TRIM(email)), '@', 2) AS email_domain
FROM customers;
MySQL uses SUBSTRING_INDEX; SQL Server may use CHARINDEX/SUBSTRING.
duplicatesMedium

Find repeated normalized emails

Normalize before grouping so formatting differences do not hide duplicates.

Pattern: GROUP BY LOWER(TRIM(email))
SELECT LOWER(TRIM(email)) AS normalized_email,
       COUNT(*) AS duplicate_count
FROM customers
GROUP BY LOWER(TRIM(email))
HAVING COUNT(*) > 1;
Define normalization rules from business requirements; over-normalization can merge distinct values.
stringMedium

Regex validation

Use a regular expression to locate invalid values when supported.

Pattern: column !~ pattern
-- PostgreSQL example
SELECT customer_id, email
FROM customers
WHERE email IS NULL
   OR email !~* '^[A-Z0-9._%+-]+@[A-Z0-9.-]+\.[A-Z]{2,}$';
Regex is useful for screening but is not a complete RFC email validator.
stringMedium

Remove non-digit characters

Normalize phone numbers using regex replacement.

Pattern: REGEXP_REPLACE(value, '[^0-9]', '', 'g')
SELECT phone,
       REGEXP_REPLACE(phone, '[^0-9]', '', 'g') AS clean_phone
FROM customers;
Regex replacement flags and syntax vary by SQL engine.
stringMedium

Find first and last token

Parse names or paths using database string functions.

Pattern: split / substring / reverse patterns
SELECT full_name,
       SPLIT_PART(TRIM(full_name), ' ', 1) AS first_token,
       REGEXP_REPLACE(TRIM(full_name), '^.*\s', '') AS last_token
FROM people;
Names are culturally complex; avoid assuming two-token structure in production systems.

08. Classic Interview Patterns

Solve recurring interview questions using ranking, grouping keys, comparisons, and sequence logic.

10 patterns
rankingEasy

Second highest distinct salary

Rank distinct salaries and keep rank two.

Pattern: DENSE_RANK() ORDER BY salary DESC
WITH ranked AS (
  SELECT employee_id,
         salary,
         DENSE_RANK() OVER (ORDER BY salary DESC) AS salary_rank
  FROM employees
)
SELECT *
FROM ranked
WHERE salary_rank = 2;
DENSE_RANK returns all employees tied at the second distinct salary.
rankingMedium

Nth highest salary

Parameterize the dense-rank pattern for any distinct salary rank.

Pattern: DENSE_RANK = :n
WITH ranked AS (
  SELECT employee_id,
         salary,
         DENSE_RANK() OVER (ORDER BY salary DESC) AS salary_rank
  FROM employees
)
SELECT *
FROM ranked
WHERE salary_rank = :n;
Clarify whether the interviewer wants the Nth row or Nth distinct value.
aggregateMedium

Top salary without window functions

Join each group to its maximum metric.

Pattern: GROUP BY group → MAX(metric) → join back
SELECT e.*
FROM employees e
JOIN (
  SELECT department, MAX(salary) AS max_salary
  FROM employees
  GROUP BY department
) m
  ON m.department = e.department
 AND m.max_salary = e.salary;
This keeps ties. Add another rule when exactly one row is required.
islandsHard

Gap and islands: consecutive integers

Consecutive values share a constant value-minus-row-number key.

Pattern: value - ROW_NUMBER() = island key
WITH grouped AS (
  SELECT seat_id,
         seat_id - ROW_NUMBER() OVER (ORDER BY seat_id) AS island_id
  FROM cinema
  WHERE free = 1
), valid AS (
  SELECT island_id
  FROM grouped
  GROUP BY island_id
  HAVING COUNT(*) >= 2
)
SELECT g.seat_id
FROM grouped g
JOIN valid v USING (island_id)
ORDER BY g.seat_id;
Use a date-minus-row-number variation for consecutive dates.
islandsHard

Longest consecutive login streak

Group consecutive login dates, then count each island.

Pattern: login_date - row_number × 1 day
WITH distinct_logins AS (
  SELECT DISTINCT user_id, login_date::date AS login_date
  FROM logins
), grouped AS (
  SELECT user_id,
         login_date,
         login_date - ROW_NUMBER() OVER (
           PARTITION BY user_id ORDER BY login_date
         ) * INTERVAL '1 day' AS island_key
  FROM distinct_logins
), streaks AS (
  SELECT user_id, island_key, COUNT(*) AS streak_days
  FROM grouped
  GROUP BY user_id, island_key
)
SELECT user_id, MAX(streak_days) AS longest_streak
FROM streaks
GROUP BY user_id;
Deduplicate same-day logins before constructing streaks.
gapsMedium

Find missing sequence numbers

Compare each value with its predecessor and expose gaps.

Pattern: current_id > previous_id + 1
WITH x AS (
  SELECT id,
         LAG(id) OVER (ORDER BY id) AS previous_id
  FROM sequence_table
)
SELECT previous_id + 1 AS gap_start,
       id - 1 AS gap_end
FROM x
WHERE id > previous_id + 1;
Generate individual missing values only when required; ranges are more scalable.
islandsHard

Consecutive events at least 3 times

Build islands and filter by island length.

Pattern: group sequence → COUNT(*) HAVING >= N
WITH numbered AS (
  SELECT user_id,
         event_date::date AS event_date,
         event_date::date - ROW_NUMBER() OVER (
           PARTITION BY user_id ORDER BY event_date::date
         ) * INTERVAL '1 day' AS grp
  FROM events
), streaks AS (
  SELECT user_id, grp, COUNT(*) AS days
  FROM numbered
  GROUP BY user_id, grp
)
SELECT user_id, days
FROM streaks
WHERE days >= 3;
Choose the sequence grain carefully: day, transaction ID, or another ordered unit.
aggregateMedium

Mode: most frequent value

Count each value and rank frequencies.

Pattern: COUNT(*) → DENSE_RANK by frequency
WITH frequencies AS (
  SELECT product_id, COUNT(*) AS purchase_count
  FROM orders
  GROUP BY product_id
), ranked AS (
  SELECT frequencies.*,
         DENSE_RANK() OVER (ORDER BY purchase_count DESC) AS rnk
  FROM frequencies
)
SELECT * FROM ranked WHERE rnk = 1;
DENSE_RANK returns all modes when frequencies tie.
windowMedium

Products above category average

Compare each detail row with a partition-level average.

Pattern: AVG(metric) OVER (PARTITION BY category)
WITH x AS (
  SELECT product_id,
         category,
         price,
         AVG(price) OVER (PARTITION BY category) AS category_avg
  FROM products
)
SELECT *
FROM x
WHERE price > category_avg;
Window functions preserve rows, unlike GROUP BY.
windowMedium

Find repeated status changes

Compare each status with the previous status and keep transition points.

Pattern: status <> LAG(status)
WITH x AS (
  SELECT account_id,
         event_time,
         status,
         LAG(status) OVER (
           PARTITION BY account_id ORDER BY event_time
         ) AS previous_status
  FROM account_events
)
SELECT *
FROM x
WHERE previous_status IS NULL
   OR status <> previous_status;
Use IS DISTINCT FROM in PostgreSQL when NULL is a meaningful status.

09. Window Logic Without Window Functions

Understand the underlying self-join logic behind LAG, LEAD, ranking, running totals, and rolling calculations.

6 patterns
self-joinMedium

LAG using self join

Match the current sequential ID with the prior ID.

Pattern: current.id = previous.id + 1
SELECT t1.id,
       t1.amount,
       t2.amount AS previous_amount
FROM transactions t1
LEFT JOIN transactions t2
  ON t1.id = t2.id + 1
ORDER BY t1.id;
This only works when IDs are contiguous. A correlated MAX lookup handles gaps.
self-joinMedium

LEAD using self join

Match the current sequential ID with the next ID.

Pattern: current.id = next.id - 1
SELECT t1.id,
       t1.amount,
       t2.amount AS next_amount
FROM transactions t1
LEFT JOIN transactions t2
  ON t1.id = t2.id - 1
ORDER BY t1.id;
As with LAG, contiguous sequence assumptions matter.
self-joinHard

Running total using self join

Join each row to all earlier rows and aggregate.

Pattern: earlier.order_key <= current.order_key
SELECT t1.order_date,
       t1.sales,
       SUM(t2.sales) AS running_total
FROM daily_sales t1
JOIN daily_sales t2
  ON t2.order_date <= t1.order_date
GROUP BY t1.order_date, t1.sales
ORDER BY t1.order_date;
This is usually O(n²) and less efficient than a window function.
self-joinHard

Rolling 30-day total using self join

Join each current date to rows inside its previous 29-day interval.

Pattern: prior_date BETWEEN current_date - 29 days AND current_date
SELECT d1.order_date,
       SUM(d2.sales) AS rolling_30_day_sales
FROM daily_sales d1
JOIN daily_sales d2
  ON d2.order_date BETWEEN d1.order_date - INTERVAL '29 days'
                       AND d1.order_date
GROUP BY d1.order_date
ORDER BY d1.order_date;
Pre-aggregate to one row per day before this join to prevent duplicate-day multiplication.
self-joinHard

ROW_NUMBER using correlated count

Count rows that sort ahead of the current row.

Pattern: 1 + COUNT(rows before current)
SELECT e1.employee_id,
       e1.salary,
       1 + (
         SELECT COUNT(*)
         FROM employees e2
         WHERE e2.salary > e1.salary
            OR (e2.salary = e1.salary
                AND e2.employee_id < e1.employee_id)
       ) AS row_num
FROM employees e1;
A unique tie-breaker is required to emulate ROW_NUMBER exactly.
self-joinHard

DENSE_RANK using correlated count

Count distinct values greater than the current value.

Pattern: 1 + COUNT(DISTINCT greater_values)
SELECT e1.employee_id,
       e1.salary,
       1 + (
         SELECT COUNT(DISTINCT e2.salary)
         FROM employees e2
         WHERE e2.salary > e1.salary
       ) AS dense_rank_num
FROM employees e1;
Use COUNT(*) rather than COUNT(DISTINCT) to model RANK-style gaps.

10. DML, Transactions & Data Integrity

Write safe INSERT, UPDATE, DELETE, UPSERT, transaction, and constraint logic.

7 patterns
dmlEasy

INSERT from SELECT

Load transformed rows from one relation into another.

Pattern: INSERT INTO target(columns) SELECT ...
INSERT INTO monthly_sales (month_start, product_id, sales)
SELECT DATE_TRUNC('month', order_date),
       product_id,
       SUM(amount)
FROM orders
GROUP BY DATE_TRUNC('month', order_date), product_id;
List target columns explicitly so schema changes do not silently break the load.
dmlMedium

Update using another table

Update target rows from a joined source.

Pattern: UPDATE target SET ... FROM source WHERE ...
UPDATE employees e
SET department_name = d.department_name
FROM departments d
WHERE d.department_id = e.department_id;
PostgreSQL syntax shown; SQL Server and MySQL use different joined-update forms.
dmlMedium

UPSERT

Insert new keys and update existing keys atomically.

Pattern: INSERT ... ON CONFLICT ... DO UPDATE
INSERT INTO customer_balance (customer_id, balance)
VALUES (101, 250.00)
ON CONFLICT (customer_id)
DO UPDATE
SET balance = EXCLUDED.balance;
PostgreSQL example. MySQL uses ON DUPLICATE KEY; SQL Server often uses MERGE or separate statements.
transactionEasy

Safe transaction

Commit a group of statements together or roll all of them back.

Pattern: BEGIN → statements → COMMIT / ROLLBACK
BEGIN;

UPDATE accounts
SET balance = balance - 100
WHERE account_id = 1;

UPDATE accounts
SET balance = balance + 100
WHERE account_id = 2;

COMMIT;
Check affected-row counts and use appropriate locking/isolation for money transfers.
constraintEasy

Primary, unique, foreign, check constraints

Push essential data rules into the database.

Pattern: PRIMARY KEY | UNIQUE | FOREIGN KEY | CHECK
CREATE TABLE employees (
  employee_id   BIGINT PRIMARY KEY,
  email         TEXT UNIQUE NOT NULL,
  department_id BIGINT REFERENCES departments(department_id),
  salary        NUMERIC(12,2) CHECK (salary >= 0),
  status        TEXT CHECK (status IN ('Active', 'Inactive'))
);
Constraints protect every writer, not only one application.
dmlMedium

Delete with EXISTS

Delete target rows that meet a relationship condition.

Pattern: DELETE FROM target WHERE EXISTS (...) 
DELETE FROM sessions s
WHERE EXISTS (
  SELECT 1
  FROM users u
  WHERE u.user_id = s.user_id
    AND u.status = 'Deleted'
);
Run the equivalent SELECT first and use a transaction for high-risk changes.
dmlHard

SCD Type 2 close-and-insert

Expire the current dimension row and insert a new version.

Pattern: UPDATE current row → INSERT new current row
BEGIN;

UPDATE customer_dim
SET valid_to = CURRENT_DATE,
    is_current = FALSE
WHERE customer_id = :customer_id
  AND is_current = TRUE;

INSERT INTO customer_dim (
  customer_id, customer_tier, valid_from, valid_to, is_current
)
VALUES (
  :customer_id, :new_tier, CURRENT_DATE, NULL, TRUE
);

COMMIT;
Production implementations need change detection, concurrency control, and idempotency.

11. Performance, Indexing & Debugging

Write sargable filters, understand index design, inspect execution plans, and prevent accidental row explosions.

8 patterns
performanceMedium

Sargable predicates

Keep indexed columns unwrapped so the optimizer can seek by range.

Pattern: column >= boundary AND column < boundary
-- Better
WHERE order_date >= DATE '2026-01-01'
  AND order_date <  DATE '2027-01-01'

-- Often worse
-- WHERE EXTRACT(YEAR FROM order_date) = 2026
Expression indexes can help, but range predicates are portable and clear.
indexMedium

Composite index column order

Order index columns around equality filters, range filters, joins, and sorting.

Pattern: equality columns → range/order columns → included columns
CREATE INDEX idx_orders_customer_date
ON orders (customer_id, order_date DESC);

SELECT *
FROM orders
WHERE customer_id = 101
  AND order_date >= DATE '2026-01-01'
ORDER BY order_date DESC;
The leftmost prefix matters; one index rarely serves every query.
indexMedium

Covering index

Include selected columns to reduce table lookups when supported.

Pattern: key columns + INCLUDE(output columns)
CREATE INDEX idx_orders_status_date
ON orders (status, order_date)
INCLUDE (customer_id, amount);
INCLUDE syntax is engine-specific; wider indexes cost more storage and write time.
performanceMedium

Read EXPLAIN plans

Inspect scan type, join strategy, row estimates, sorting, and actual timing.

Pattern: EXPLAIN (ANALYZE, BUFFERS) query
EXPLAIN (ANALYZE, BUFFERS)
SELECT customer_id, SUM(amount)
FROM orders
WHERE order_date >= CURRENT_DATE - INTERVAL '30 days'
GROUP BY customer_id;
Use ANALYZE carefully in production because it actually executes the query.
performanceEasy

Avoid SELECT *

Read only required columns to reduce I/O and make contracts explicit.

Pattern: SELECT required_columns
SELECT order_id,
       customer_id,
       order_date,
       amount
FROM orders
WHERE status = 'Open';
Column pruning matters especially for wide tables and columnar storage.
debugMedium

Detect accidental many-to-many joins

Compare row counts and key uniqueness before and after joining.

Pattern: validate keys → join → validate row count
SELECT customer_id, COUNT(*)
FROM customers
GROUP BY customer_id
HAVING COUNT(*) > 1;

SELECT customer_id, COUNT(*)
FROM customer_status
GROUP BY customer_id
HAVING COUNT(*) > 1;
A join is not wrong merely because rows multiply; the multiplication must match the intended grain.
debugEasy

COUNT(*) vs COUNT(column)

Understand that COUNT(column) skips NULL values.

Pattern: COUNT(*) counts rows; COUNT(col) counts non-NULL values
SELECT COUNT(*) AS total_rows,
       COUNT(manager_id) AS rows_with_manager,
       COUNT(DISTINCT department_id) AS distinct_departments
FROM employees;
COUNT(1) is generally equivalent to COUNT(*) in modern optimizers.
performanceMedium

CTE vs temporary table

Choose a logical query stage or a materialized reusable intermediate result.

Pattern: CTE for readability; temp table for reuse/indexing/stats
CREATE TEMP TABLE recent_orders AS
SELECT *
FROM orders
WHERE order_date >= CURRENT_DATE - INTERVAL '90 days';

CREATE INDEX ON recent_orders (customer_id);
Optimizer behavior for CTE materialization differs by engine and version.

12. Rapid-Fire Differences

Compact comparisons for common SQL theory questions asked before or after coding rounds.

8 patterns
differenceEasy

WHERE vs HAVING

WHERE filters rows before grouping; HAVING filters grouped results.

Pattern: WHERE → GROUP BY → HAVING
-- Row filter
WHERE status = 'Active'

-- Group filter
HAVING COUNT(*) >= 5
Aggregate expressions belong in HAVING unless computed in an earlier query stage.
differenceEasy

DELETE vs TRUNCATE vs DROP

DELETE removes selected rows; TRUNCATE removes all rows efficiently; DROP removes the object.

Pattern: rows | all rows | object
DELETE FROM logs WHERE log_date < DATE '2025-01-01';
TRUNCATE TABLE staging_orders;
DROP TABLE obsolete_table;
Transaction support, identity reset, logging, and trigger behavior vary by database.
differenceEasy

PRIMARY KEY vs UNIQUE

Both enforce uniqueness, but a primary key identifies the row and is implicitly NOT NULL.

Pattern: one primary key; multiple unique constraints
CREATE TABLE users (
  user_id BIGINT PRIMARY KEY,
  email TEXT UNIQUE NOT NULL
);
A composite primary key can contain multiple columns but remains one primary-key constraint.
differenceMedium

INNER JOIN vs EXISTS

JOIN returns combined rows; EXISTS checks whether at least one match exists.

Pattern: return columns vs test relationship
SELECT c.*
FROM customers c
WHERE EXISTS (
  SELECT 1 FROM orders o
  WHERE o.customer_id = c.customer_id
);
A JOIN may duplicate parent rows when multiple child rows match.
differenceEasy

ROW_NUMBER vs RANK vs DENSE_RANK

Unique sequence, rank with gaps, and rank without gaps.

Pattern: 1,2,3 | 1,2,2,4 | 1,2,2,3
ROW_NUMBER() OVER (ORDER BY score DESC)
RANK()       OVER (ORDER BY score DESC)
DENSE_RANK() OVER (ORDER BY score DESC)
Clarify tie requirements before choosing a function.
differenceHard

ROWS vs RANGE window frames

ROWS counts physical rows; RANGE groups peers or value ranges depending on syntax.

Pattern: physical positions vs ORDER BY value semantics
SUM(amount) OVER (
  ORDER BY event_time
  ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
)
Use ROWS for predictable row counts; use RANGE for value/time intervals when supported.
differenceMedium

View vs materialized view

A view stores a query; a materialized view stores query results and must be refreshed.

Pattern: virtual result vs persisted result
CREATE VIEW active_customers AS
SELECT * FROM customers WHERE status = 'Active';

CREATE MATERIALIZED VIEW monthly_sales AS
SELECT DATE_TRUNC('month', order_date), SUM(amount)
FROM orders
GROUP BY 1;
Materialized-view syntax and refresh strategies are engine-specific.
differenceMedium

Clustered vs nonclustered index

A clustered organization controls row storage order; secondary indexes store separate lookup structures.

Pattern: table organization vs separate access path
-- SQL Server-style terminology varies by database.
CREATE CLUSTERED INDEX cx_orders_date
ON orders(order_date);

CREATE NONCLUSTERED INDEX ix_orders_customer
ON orders(customer_id);
PostgreSQL, MySQL/InnoDB, and SQL Server implement physical organization differently.
